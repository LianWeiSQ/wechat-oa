## 账号定位

把 AI Agent 从“模型热闹”拉回“工程上线”。账号要长期占住一个清晰心智：

> 面向技术负责人、AI 工程师、创业团队和企业 AI 采购方，拆解 Agent 真正落地时的状态管理、工具编排、可靠性、成本和组织协作问题。

公众号承担“深度信任”和“体系化内容资产”，小红书承担“发现流增长”和“轻量观点种草”。两边不是重复搬运，而是同一主题的不同颗粒度表达。

## 目标读者

1. AI Agent / AI Infra 工程师：关心框架选型、状态恢复、工具调用、评测、生产稳定性。
2. 技术负责人 / CTO：关心 Agent 项目为什么难上线、怎么分阶段投入、团队如何组织。
3. 企业老板 / AI 咨询客户：关心 AI 项目是否能真正降本增效，如何判断供应商是不是只会讲模型。
4. 想转 Agent 方向的技术人：关心学习路径、项目作品、面试表达和工程判断力。

## 小红书视觉与文案规则

- 封面优先 9:16，主标题 8-14 个字，副标题补充场景。
- 视觉保持“技术白板 + 工程杂志”风格：干净背景、低饱和色、少装饰、重点词高亮。
- 每篇正文控制在 450-900 字，段落短，第一屏必须有冲突。
- 每篇只讲一个判断，不把公众号长文全部塞进去。
- CTA 优先“收藏/评论关键词/去公众号看长文”，少用强销售表达。

## 新主题表达：面试焦虑 + 技术深度

更新判断：Harness / Agent 工程化内容可以适度引入面试焦虑，因为目标读者正在从“会调模型”进入“要证明自己懂生产级 Agent”的阶段。焦虑入口可以提高点击，但正文必须给出真实技术增量，不能停留在标题党。

推荐表达公式：

面试问题 -> 常见错误回答 -> 背后的工程本质 -> 正确回答框架 -> 项目落地证明。

适合标题方向：

1. 面试官问 Agent 断点续跑，你还只会说保存聊天记录吗？
2. Agent 平台面试，真正拉开差距的是 Harness
3. 为什么会讲 ThreadState 的工程师，更像做过生产级 Agent
4. 只会 RAG 和提示词，面试 Agent Infra 会越来越吃亏
5. Agent 长任务一中断就重跑，面试官基本知道你没做过生产系统

写作原则：

- 焦虑来自真实岗位变化，不编造夸张战绩和虚假大厂案例。
- 开头可以强冲突，正文必须落到状态、幂等、checkpoint、恢复策略、观测和人工介入。
- 专业词要出现，但每个专业词都要翻译成“面试官到底在考什么”。
- 文章结尾要给读者一套可背、可改、可写进项目经历的回答框架。
- 原始源码文章作为素材库，不直接复述源码，不搬运大段原文。

## 首周评论回复模板

读者问“Harness 是不是工作流引擎？”

> 有重叠，但不一样。工作流更偏步骤编排，Harness 还要管会话状态、工具副作用、失败恢复和多入口执行模型。我后面会单独写一篇对比。

读者问“我们只是 Demo，要不要管这么复杂？”

> Demo 阶段可以先简化，但至少要留出状态源和恢复点。不然后面从 Demo 迁到生产时，很多逻辑要推倒重来。

读者问“有没有自检清单？”

> 有，我正在把 Harness 系列整理成清单。你可以先看 4 个问题：状态存哪里、工具是否幂等、失败状态是否细分、能不能断点续跑。

读者反驳“模型强了不就解决了吗？”

> 模型强会提高推理质量，但不会自动解决外部系统副作用。比如邮件是否已发送、订单是否已创建、任务是否已落盘，这些都必须由 Harness 记录。
