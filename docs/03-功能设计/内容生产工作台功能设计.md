# 内容生产工作台功能设计

这篇文档合并专业长文生成、技术文章库与公众号工作台拆分、前端编辑台视觉整改三部分，作为当前产品功能设计的统一入口。

## 专业公众号长文生成设计

Date: 2026-05-14

## Goal

Upgrade `wechat-oa` from “article analysis creates a simple draft” to a local-first WeChat article production workflow that can generate professional, readable, low-AI-flavor long-form drafts for AI topics, with generated supporting images inserted into the draft.

The chosen product direction is the mixed style: use a Harness-like engineering backbone for credibility, then adapt the presentation to WeChat reading habits.

## Sources Checked

- OpenAI GPT Image 2 model page: `https://developers.openai.com/api/docs/models/gpt-image-2`
- OpenAI Images API reference: `https://developers.openai.com/api/reference/resources/images`

The design uses `gpt-image-2` with `POST /v1/images/generations`, stores returned image bytes locally, and keeps text generation on the existing OpenAI-compatible Codex bridge.

## Non-Goals

- Do not build public publishing or multi-user collaboration.
- Do not copy external article images into the local library.
- Do not make generated images mandatory for exporting a text draft.
- Do not route image generation through the local Codex chat bridge in v1; image generation uses the OpenAI Images API directly.
- Do not attempt automatic WeChat media upload for every generated image in this pass. Existing WeChat draft delivery can continue to use the current `defaultThumbMediaId` fallback until image upload support is added separately.

## User Workflow

1. User imports or selects an article in the local workbench.
2. User runs the existing AI analysis template to get reusable angles and viral-scoring context.
3. User clicks a new “生成专业长文 + 配图” action.
4. The app asks the text model to produce a structured long-form article draft:
   - title
   - subtitle or deck
   - opening hook
   - HTML body sections
   - pull quotes
   - image briefs and image prompts
5. The app calls `gpt-image-2` for the generated image prompts:
   - one WeChat-cover-style hero image
   - one in-body explanatory image, usually architecture, workflow, risk map, or decision matrix
6. Generated images are saved under local app data and inserted into the HTML draft as `<figure>` blocks with `img`, `alt`, and `figcaption`.
7. The draft remains usable if image generation partially fails. Failed image slots show status and the original prompt so the user can retry or manually regenerate.

## Writing Style

The writer should borrow the discipline of a strong engineering blog without imitating or copying any specific Harness content:

- Start from a concrete production problem, not a broad trend claim.
- Explain the real operating constraints: latency, cost, permissions, reliability, observability, rollback, migration cost, and organizational ownership.
- Prefer architecture and trade-off language over hype language.
- Include failure modes and “where this breaks” sections.
- Translate engineering details into business consequences for non-engineering readers.
- Use short paragraphs suitable for WeChat reading.
- Avoid boilerplate transitions such as “首先”, “其次”, “综上”, “值得注意的是”, “在当今时代”, and generic listicle phrasing.
- Prefer scenario, evidence, contrast, and actionable judgment over abstract praise.

## Draft Structure

The generated article should follow this default shape:

1. Title: sharp but not clickbait.
2. Deck: one sentence explaining why the topic matters now.
3. Opening scene: a concrete scenario or tension in the first 300 Chinese characters.
4. Core judgment: the main argument in one memorable sentence.
5. Engineering body:
   - problem statement
   - production constraints
   - architecture or operating model
   - trade-offs and failure modes
   - what changes for developers, teams, and business owners
6. Practical conclusion:
   - what to do now
   - what not to overbuild
   - what signal to watch next

## Image Strategy

Two images are generated by default:

- Hero image:
  - purpose: cover and article identity
  - style: clean technical magazine, professional, high signal, no fake UI chrome, no dense text
  - default size: `1536x1024`
- Explanatory image:
  - purpose: clarify one non-obvious concept from the article
  - style: architecture diagram, workflow map, risk matrix, or decision map
  - default placement: after the first engineering section

Each image asset stores:

- draft id
- role: `hero` or `explanation`
- generation status: `pending`, `generated`, or `failed`
- local file path
- public route path
- prompt
- revised prompt, if returned by the API
- alt text
- caption
- model metadata
- error message, if any
- created and updated timestamps

## Data Model

Add `DraftImageAsset`:

- `id`
- `draftId`
- `role`
- `status`
- `localPath`
- `publicPath`
- `prompt`
- `revisedPrompt`
- `alt`
- `caption`
- `model`
- `size`
- `error`
- `createdAt`
- `updatedAt`

Extend settings with `ImageSettings`:

- `baseUrl`: default `https://api.openai.com/v1`
- `apiKey`: from saved encrypted value or `OPENAI_API_KEY`
- `model`: default `gpt-image-2`
- `size`: default supported landscape size chosen by implementation

The existing `settings` table can store this as another JSON setting key. Secrets should use the existing local AES-GCM helper pattern.

## API Design

Add or extend routes:

- `POST /api/articles/:id/professional-draft`
  - input: `analysisRunId?`, `imageMode?`
  - behavior: generate article draft, create local draft, generate images, insert image figures, return draft and image assets
- `GET /api/settings/image`
  - returns public image settings, masking the API key
- `PUT /api/settings/image`
  - saves image base URL, model, size, and encrypted API key
- `GET /api/assets/images/:file`
  - serves locally generated images from the app data directory
- optional follow-up: `POST /api/drafts/:id/images/:assetId/regenerate`
  - regenerates a failed or stale image slot

## Error Handling

- Missing text model settings: return a clear setup error and do not create a draft.
- Text generation failure: do not create a draft.
- Invalid writer JSON: recover if possible; otherwise return a structured error.
- Missing image key: create the draft and image slots with `failed` status and an actionable message.
- Image API failure: create or keep the draft; mark only the affected asset as `failed`.
- Local file write failure: mark asset as `failed`; keep draft text.
- Asset route traversal attempt: reject any filename containing path separators or unsafe characters.

## UI Design

In the workbench detail pane:

- Keep “生成拆解” as the analysis action.
- Add “生成专业长文 + 配图” near the current draft/analysis controls.
- Show progress states:
  - writing article
  - generating hero image
  - generating explanation image
  - saved
- Show generated draft preview using existing draft body rendering.
- Show image asset cards with:
  - role
  - status
  - preview
  - caption
  - error, if failed
  - prompt copy text

In settings:

- Keep current AI settings for text.
- Add image settings:
  - base URL
  - model
  - size
  - API key
  - saved/has-key status

## Testing

Add focused tests for:

- article writer prompt contains Harness-style engineering constraints and anti-AI-flavor rules
- writer parses structured output into title, body, and image briefs
- malformed writer output returns a clear error
- image settings mask the API key in public responses
- image asset store persists generated and failed assets
- image generator decodes `b64_json` and writes a local image file
- professional-draft route creates a draft even when one image generation fails
- workbench shows the new action and progress states

## Rollout

This is a local-first feature. The first implementation should be functional and testable without making live model calls by using injected fetchers in tests. Live API calls are verified manually only when credentials are available.

Generated image assets live under `data/generated-images`, which remains ignored by git.

Verification completed:

- `pnpm test`
- `pnpm lint`
- `pnpm build`
- browser smoke check on `http://127.0.0.1:3002`

## Implementation Notes

Implemented modules:

- `src/lib/article-writer.ts`: builds the professional writing prompt, calls the existing OpenAI-compatible text model path, and parses structured draft JSON.
- `src/lib/image-generation.ts`: calls `POST /v1/images/generations`, decodes GPT image `b64_json`, and writes PNG files under `data/generated-images`.
- `src/lib/professional-drafts.ts`: orchestrates draft creation, image generation, successful figure insertion, and failed-image prompt preservation.
- `src/lib/draft-images.ts`: persists generated image asset metadata in SQLite.
- `src/app/api/articles/[id]/professional-draft/route.ts`: exposes the workbench generation action.
- `src/app/api/settings/image/route.ts`: exposes public image settings and encrypted API key saving.
- `src/app/api/assets/images/[file]/route.ts`: serves generated local image files with filename validation.
- `src/components/workbench.tsx`: adds professional draft generation, image asset status cards, and image model settings.

## 技术文章库与微信公众号工作台拆分

Date: 2026-05-15

## Summary

`wechat-oa` 现在拆成两个工作区：

- `技术文章库`：保存任意技术文章，支持手动粘贴和 URL 解析，负责智能处理、文章质量判断和多视角分析。
- `微信公众号`：引用文章库内容生成公众号长文、配图、导出和微信草稿投递，不再承担文章导入职责。

## Implemented Boundaries

- 文章库 owns：`/api/library/import/*`、`/api/library/articles/*`、Article、ArticleParseRun、ContentAgentRun。
- 微信公众号 owns：`/api/wechat/articles/:id/professional-draft`、草稿、配图资产、微信配置和投递。
- 共享：AI settings、Image settings、AnalysisRun、TopicCandidate。

## Data Model Notes

`Article` 已泛化为通用技术文章：

- 新字段：`sourceType`, `sourceName`, `contentHtml`, `contentText`
- 兼容字段：`sourceAccount`, `content`
- 兼容迁移：已有 `source_account/content` 数据会映射到新字段，不删除旧列。

新增记录：

- `ArticleParseRun`：保存 URL 解析策略、质量分、元数据和 fallback 原因。
- `ContentAgentRun`：保存 Agent 状态、步骤、文章类型、质量分、推荐模板和建议动作。

## Agent Behavior

内容处理 Agent 使用当前本地 OpenAI-compatible 模型配置，默认仍是：

- `baseUrl`: `http://127.0.0.1:3000`
- `model`: `gpt-5.4`
- `wireApi`: `responses`

AI 网关配置以本地 `.env.local` 为权威来源。只要存在 `OPENAI_*` 环境变量，应用不会再从 SQL `settings.ai` 读取模型配置，也不会把模型配置保存回 SQL。

Agent 输出：

- 文章类型
- 质量评分
- 推荐分析模板
- 建议动作
- 可追溯步骤日志

缺少模型配置时，Agent 保存 failed run，并给出可操作错误，不破坏文章数据。

## 前端编辑台视觉整改

Date: 2026-05-15

## Summary

本轮使用 `frontend-design` skill 的原则整改 `wechat-oa` 前端：避免默认后台和模板化 AI 界面，把产品气质调整成 Hugging Face 技术指南感的黑金 AI 内容工作台。

## Design Direction

- 基调：暗色技术指南 + 编辑部工作台，而不是营销页或普通 SaaS dashboard。
- 视觉：黑色/石墨底、顶部金橙强调线、深色半透明面板、柔边框、金橙主按钮。
- 字体：标题和阅读区保留中文宋体气质，控件使用更干净的中文无衬线。
- 布局：保留三栏生产流，左侧文章库，中间阅读和智能处理，右侧导入或公众号配置。
- 边界：不动后端能力，不新增依赖，不引入图片资产或营销 hero。

## Implemented

- 重写全局视觉 tokens 和基础控件样式为黑金主题。
- 将 `Workbench` 从大量行内 Tailwind 色值改为语义化 CSS 类。
- 顶部工作区切换改成更明确的 segmented control。
- 左侧文章行增加来源类型 chip，提升文章库语义。
- 阅读区改成更适合长文的排版和留白。
- Agent、拆解、候选角度、草稿、导入、配置统一为同一套面板系统。
- 缩短空状态文案，降低说明书感。
- 修复 390px 移动端工作区切换横向溢出。
- 修复导入文章阅读器：展示前移除源站内联样式、脚本、iframe、远程图片和纯 URL 图占位，避免黑字压暗底、破图链接漂浮、外部样式污染本地阅读体验。
- 阅读区在中等宽度下优先获得完整列宽，智能处理面板下移，避免长标题和正文被右侧面板挤窄。
- 右侧新增文章入口改为黑金 segmented control 和状态反馈，避免亮色默认控件破坏整体主题。
