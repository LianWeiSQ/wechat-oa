# Agent 上线前 Harness 自检清单

适用对象：Agent 平台研发、AI Infra 工程师、技术负责人、准备把 Agent Demo 推向生产的团队。
使用方式：每个问题按 `已确认 / 待补齐 / 不适用` 标记。只要出现多个“待补齐”，就先不要急着上线。

## 1. 状态管理

1. 这个 Agent 有没有唯一可信状态源，比如 `ThreadState`、`RunState` 或 `TaskState`？
2. 任务 ID、用户 ID、会话 ID、run ID 的关系是否清楚？
3. 当前步骤、已完成步骤、失败步骤是否结构化保存？
4. 模型输出、工具调用结果、用户确认信息是否写入同一条执行链路？
5. 状态是只存在内存里，还是有可靠持久化？
6. 状态更新失败时，系统会报错、重试，还是静默丢失？

## 2. 工具调用

1. 每次工具调用是否有唯一请求 ID？
2. 工具返回结果是否有结构化字段，而不是只保存原始文本？
3. 工具调用成功后，外部结果 ID 是否写入状态？
4. 失败原因是否区分超时、权限、参数错误、外部系统异常？
5. 工具是否声明哪些操作有外部副作用？
6. 工具调用日志是否能追溯到具体 run ID 和步骤 ID？

## 3. 幂等与副作用

1. 创建订单、工单、邮件、文件、数据库写入等动作是否有幂等键？
2. 任务恢复后，系统能否识别某个外部动作已经执行过？
3. 重试策略是否区分“可安全重试”和“需要先对账”？
4. 状态未知时，是否有外部查询接口确认结果？
5. 重复执行会不会造成重复扣款、重复通知、重复创建资源？
6. 人工介入后，系统状态是否会被同步更新？

## 4. Checkpoint 与恢复

1. 系统在哪些节点写 checkpoint？
2. checkpoint 是按时间写，还是按关键事实写？
3. 进程重启后，能否从最后一个稳定 checkpoint 恢复？
4. 恢复时是否会先检查未完成工具调用？
5. 恢复逻辑是否区分 `completed`、`failed`、`timed_out`、`unknown`？
6. 有没有实际做过“执行到一半杀进程，再重启恢复”的演练？

## 5. 执行循环

1. Agent Loop 是否明确包含 Think、Act、Observe、Repeat？
2. Observe 阶段是否把外部结果写回状态？
3. 模型是否只能基于当前可信状态继续推理？
4. 是否允许模型绕过 Harness 直接决定外部动作？
5. 长任务是否支持异步执行和进度查询？
6. 用户中途取消、修改目标或补充信息时，状态如何更新？

## 6. 观测与人工介入

1. 每次 run 是否有可读的执行轨迹？
2. 失败时能否看到失败步骤、失败原因和当前状态？
3. 是否有人工接管入口？
4. 人工接管后，Agent 是否知道哪些动作已经由人完成？
5. 是否有降级方案，比如转人工、暂停、只读模式？
6. 发布前是否有最小验收标准和回滚策略？

## 7. 发布前最低标准

上线前至少跑通这 5 个场景：

1. 正常完成一个端到端任务。
2. 工具调用超时后安全重试。
3. 外部动作成功但响应丢失后先对账再继续。
4. 进程中断后从 checkpoint 恢复。
5. 人工介入后状态仍然一致。

如果这 5 个场景跑不通，问题通常不在模型，而在 Harness。
