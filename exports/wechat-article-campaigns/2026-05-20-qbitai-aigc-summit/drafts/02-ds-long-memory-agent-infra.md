<h1>长期记忆会成为 Agent 的下一层基础设施吗？</h1><p>现在的 Agent 很擅长打单次战役，但一旦让它记住你三周前随口提过的预算数字，或者延续两个月前那个被打断的流程，它就露馅了。长上下文窗口一直被当作解药，200K、1M token 的模型不断刷新纪录，可现实中的 Agent 依然会“失忆”——它忘了你是开发环境还是生产环境，忘了上次的决策边界，甚至在多轮对话里重复询问同一个配置。</p><p>这种断裂正在催生一种新的主张：<strong>长期记忆不应该只是上下文窗口的线性延长，而需要成为独立的基础设施层</strong>。量子位 2026 中国 AIGC 产业峰会的议程显示，这一层已经走到台前。EverMind CEO 邓亚峰将分享“长期记忆驱动的自进化”，探讨 AI 如何从工具演化为可持续成长的数字生产力系统；ColaOS & MarsWave CEO 冯雷则会在 Agent 圆桌中，与行业讨论 Agent 产品形态、多智能体框架与自进化引擎。两者都指向同一个命题：记忆，可能正在成为工具和智能体之间的分界线。</p><p>但概念之下，是大量尚待澄清的工程问题。记忆如何写入、检索、更新、过期？长期记忆系统会怎样影响 Agent 的一致性、个性化和安全边界？我们离可治理的记忆基础设施还有多远？</p><h2>1. 上下文窗口越来越大，为什么还需要长期记忆？</h2><p>上下文窗口的膨胀给了开发者一种错觉：只要把历史对话、文档全塞进去，Agent 就记住了。但这带来了三重矛盾。</p><p>第一，信息的密度与优先级。无限长的上下文里，Agent 往往找不到真正重要的东西——它会平均关注每一段文字，而你上周特意强调的“数据库连接串别改”可能被淹没。</p><p>第二，成本与延迟。超长上下文导致推理开销指数级上升，对需要快速响应的 Agent 场景并不友好。</p><p>第三，也是最被忽略的一点：<strong>上下文本身没有“状态”</strong>。它只是一个瞬时输入，用完即弃。而真正的记忆需要能够跨会话、跨任务甚至跨睡眠周期持续存在，并在合适的时机被主动触发。EverMind 官网将 EverOS 描述为面向 AI 智能体的 Memory OS，强调“跨时间、跨会话、跨平台保持上下文，并支持持久、主动、自我演进的记忆”。这正好戳中了上下文窗口的局限——它只是一个会话缓存，不是记忆系统。</p><blockquote>“记忆是工具和智能体之间的分界线。”——EverMind 公开资料</blockquote><p>这句话暗示了一个根本性转变：没有长期记忆的 Agent，本质上仍然是一次性的工具；只有能够积累、组织和进化记忆的智能体，才可能发展出连续的行为模式和个性。</p><h2>2. 记忆不只是“存下来”：写入、检索、更新、过期</h2><p>把长期记忆从营销概念拆成工程模块，至少要解决四个环节。</p><p><strong>写入——什么值得记？</strong>不是所有对话都同等重要。需要一套机制来判断信息的长期价值，这通常涉及重要性评分、意图识别或明确的“记住这个”指令。写入的时机也很关键：是实时流式存储，还是在会话结束时做总结摘要？不同方案会直接影响记忆的粒度和一致性。</p><p><strong>检索——如何在需要时找到？</strong>原始关键词匹配会漏掉大量语义关联。向量检索是当前主流，但单纯依赖向量又会丢失结构和时效性。更可行的方案是混合检索，结合元数据过滤、时间衰减、标签图谱，甚至将记忆组织成带时间戳的知识图谱片段。</p><p><strong>更新与冲突解决——事实变了怎么办？</strong>一个人换了手机号，Agent 要能覆盖旧记忆，而不是新旧并存造成决策混乱。这里涉及版本控制、置信度判断和来源追踪。当用户的表述和 Agent 的推断冲突时，需要有明确的解决策略，甚至允许人工“记忆修正”。</p><p><strong>过期与遗忘——无用的记忆如何淘汰？</strong>不是所有记忆都该永存。部分信息具有时间敏感性（如临时权限），部分因长久不使用而应当衰减。这类似于计算机缓存中的 LRU 策略，但需要更细粒度的生命周期管理，并与隐私合规中的“遗忘权”衔接。</p><p>这几个模块目前行业还处于早期探索阶段。EverMind 在公开资料中提到了“持久、主动、自我演进的记忆”，但具体如何落地，仍有待会后披露更多架构细节。</p><h2>3. 从 EverMind 到 ColaOS：记忆正在成为独立层</h2><p>量子位峰会上的两个项目恰好从不同方向逼近了同一目标。EverMind 走的是纵向 Memory OS 路线，试图为 Agent 提供一套脱离具体应用、可跨平台运行的记忆子系统。它的核心主张很清晰：<strong>记忆应该从 Agent 逻辑中解耦出来，成为像文件系统一样的操作系统级服务</strong>。</p><p>ColaOS / MarsWave 则从 Agent OS 的角度切入，其公开页面将产品描述为“first Soulful Agent”与“AI OS that grows with you”，强调“记忆、主动执行、上下文连续性和更少工具切换”。从字面看，他们意图构建一个以用户为中心的 AI 工作环境，记忆只是其中一环，但却是灵魂所在。圆桌讨论预计会触及 Agent 产品形态、操作系统与自进化引擎等方向，将这些问题推向前台。</p><p>这两条路径的共同点是：都不再把记忆看作模型能力的一部分，而是作为独立的基础设施来设计。这意味着它可以有自己的 API、存储后端、访问控制策略，甚至跨 Agent 共享。这种分层解耦，会深刻改变上层应用的结构。</p><h2>4. 长期记忆如何改变 Agent 的安全与治理边界？</h2><p>一旦记忆成为独立层，安全模型也要随之改变。当前 LLM 应用中，上下文安全主要通过提示词防护和输入输出过滤实现。但长期记忆引入了一个新的攻击面：<strong>历史记忆被污染</strong>。</p><p>设想一个有恶意用户，在三个月时间里逐步向 Agent 的长期记忆注入伪造的偏好或错误的事实，然后再利用这些错误记忆诱导决策。这比单次提示注入更隐蔽，也更难审计。因此，记忆的完整性校验、来源签名、不可篡改的追加日志将成为刚需。</p><p>此外，治理问题突出。企业场景中，谁有权查看、修改或删除 Agent 的记忆？记忆的保留策略如何与数据合规（如 GDPR 的删除权）对齐？如果一名员工离开公司，其个人助理的记忆该如何处置？这些问题没有一个现有 LLM 框架能完整回答。长期记忆层必须内建类似数据库的权限模型，并提供对记忆内容的可审计能力。</p><h2>5. 状态源与上下文连续性：工程上最难的部分</h2><p>我们曾在“Harness”系列中讨论过状态源的概念。状态源是 Agent 做出决策时的依据集，它横跨环境变量、配置文件、实时数据、用户设置和之前的推理结果。长期记忆本质上是状态源里最“软”也最动态的一部分，但它的管理难度远高于其他部分。</p><p>上下文连续性是表象，本质是<strong>状态源在时间轴上的跨会话一致性</strong>。Agent 在一个月前依据某个版本的记忆做出了决策，今天记忆已被更新，但那个历史决策仍是有效的。如果需要回溯或复盘，必须能准确还原当时的状态源切片。这要求记忆系统具备时间旅行查询能力，而目前鲜有产品实现。</p><p>另一个棘手问题是主动记忆与实时上下文的关系。EverMind 强调“主动”记忆，即记忆层可以不经用户提问，自行推断出应该被“记起”的信息，推送给下游决策。但如果推送的时机不当，可能干扰当前的对话流；如果推送内容有误，会导致 Agent 在不经意间表现出“幻觉”。主动记忆的触发策略，需要在工程上找到精妙的平衡。</p><h2>6. 落地前还需回答的三个问题</h2><p>尽管方向已现，但长期记忆要沉淀为可靠的基础设施，至少还有三个问题悬而未决。</p><p><strong>评测标准缺失。</strong> 如何衡量一个记忆系统的好坏？是检索命中率、遗忘及时性、污染恢复能力，还是用户感受到的“体贴度”？行业需要一个可量化的基准测试集，目前几乎是空白。没有评测，选型就退化为 PPT 比拼。</p><p><strong>与推理层的接口模糊。</strong> 记忆层究竟应该在推理前注入（pre-prompt），还是在推理过程中通过函数调用按需拉取？前者简单粗暴，但容易泄漏无关记忆；后者更精准，却要求推理引擎能感知记忆层的存在，接口定义复杂度陡增。标准化工作尚未启动。</p><p><strong>记忆的自我进化是否安全？</strong> 自进化是 EverMind 宣传的核心之一。如果 Agent 能基于自身经验改写记忆规律，甚至调整记忆的权重策略，这固然强大，但也会引发不可预测性。进化中的边界控制、人类可干预、回滚机制，都是必须前置设计的。</p><p>这些问题的答案，不会出现在任何一场峰会的 keynote 里，只会在接下来的工程实践中被慢慢闯出来。而对当下选择观望还是入场的企业来说，最重要的一步或许是：<strong>把记忆先当成一个独立模块来对待，而不是继续把它交给模型上下文去消化</strong>。哪怕从最简单的用户档案表开始，也比把一切塞进超长 prompt 更接近正确的方向。</p>
