# 面试官问 Agent 断点续跑，你还只会说保存聊天记录吗？

摘要：

越来越多 Agent 面试题，已经不是问你“会不会调模型”，而是问你“任务跑到一半挂了，怎么恢复”。如果你的回答还停留在保存聊天记录，很容易暴露一个问题：你理解的是对话机器人，不是生产级 Agent 系统。

封面 Brief：

技术面试场景，一边是候选人写着“保存聊天记录”，另一边白板上写着 checkpoint、state、idempotency、resume。风格偏技术杂志封面，紧张但克制。

## 这类题，已经开始变常见了

以前面试 Agent，很多人准备的是模型选型、RAG、提示词、LangChain。

现在面试官开始往下问：

一个 Agent 正在执行长任务，已经调用了两个外部工具，第三步网络超时，进程重启了。用户回来以后，系统怎么知道从哪里继续？

这个问题一出来，很多回答会瞬间露底。

有人说，把聊天记录存下来，再让模型继续生成。

有人说，失败了就重新跑一遍。

有人说，加一个 try-catch，异常的时候重试。

这些回答不是完全没用，但都不够。

因为面试官真正问的不是“你会不会保存上下文”，而是：

你知不知道 Agent 从 Demo 走向生产以后，最怕的不是答错一句话，而是执行状态乱掉。

## 断点续跑不是恢复聊天，是恢复执行现场

这是第一个必须讲清楚的点。

聊天记录只能说明用户和模型说过什么。

但生产级 Agent 真正要恢复的是执行现场。

它要知道：

- 这个任务的目标是什么。
- 当前执行到第几步。
- 哪些步骤已经成功。
- 哪些工具已经调用过。
- 哪些工具调用产生了外部副作用。
- 哪些失败可以重试。
- 哪些状态未知，必须先确认。
- 下一步应该继续、跳过、重试，还是交给人工处理。

如果这些信息没有结构化保存，只保存聊天记录，本质上只是让模型“凭感觉接着聊”。

而生产系统最怕的就是凭感觉。

## 面试官想听到的第一个关键词：唯一状态源

如果要把这个问题讲深，第一句话可以这样说：

Agent 断点续跑的核心不是保存对话，而是维护一个唯一可信的任务状态源。

你可以把它叫 `ThreadState`，也可以叫 `TaskState`、`RunState`、`AppState`。

名字不重要。

重要的是，Agent 执行过程里的关键事实，必须收敛到一个明确的状态对象里。

比如：

- 任务 ID。
- 当前阶段。
- 已完成步骤。
- 工具调用记录。
- 子任务状态。
- 用户确认信息。
- 外部系统返回结果。
- 恢复时需要用到的上下文。

如果这些状态散落在内存、日志、聊天记录、工具返回值里，系统一旦中断，就很难恢复。

你不知道哪个事实是真的，也不知道应该信哪一份。

这就是很多 Agent Demo 看起来很顺，一上线就崩的原因。

Demo 只需要跑通一次。

生产系统需要在失败之后还能继续。

## 第二个关键词：checkpoint 不是随便存一下

很多人听到 checkpoint，会以为就是隔一段时间保存一次。

这也不够。

真正有价值的 checkpoint，应该落在关键事实发生之后。

比如：

- Agent 已经生成稳定计划。
- 某个工具调用已经成功。
- 用户已经确认某个动作。
- 外部系统已经创建订单、工单、邮件、文件。
- 一个子任务已经完成。

这些节点一旦发生，就不能只存在模型上下文里。

因为模型上下文不是可靠存储。

它会被截断，会被压缩，也会因为下一轮生成而变得模糊。

所以 checkpoint 的本质，是把“已经发生的事实”写入可靠存储。

不是为了好看。

是为了中断之后，系统可以判断：

这一步已经发生了，不能重复做。

这一步没有发生，可以继续做。

这一步状态未知，必须先查清楚。

## 最能拉开差距的点：状态未知

面试里真正能区分水平的，不是成功和失败。

是状态未知。

举个例子。

Agent 调用外部系统创建工单。

请求已经发出去了，但响应还没回来，网络断了。

这个时候系统重启。

工单到底有没有创建成功？

不知道。

如果你直接重试，可能创建两个工单。

如果你直接跳过，可能漏掉一个工单。

如果你直接让模型判断，模型也不知道真实世界发生了什么。

这个时候正确的做法，是先做状态对账。

用幂等 ID、业务唯一键，或者外部系统查询接口，确认这个动作到底有没有发生。

确认之后，再决定继续、跳过、重试或者人工介入。

这就是为什么 Agent 工程不是“模型会思考”就够了。

因为模型不能替你保证外部世界的一致性。

## 子任务状态，也不能只靠感觉

很多复杂 Agent 不是一个模型从头跑到尾，而是 Lead Agent 拆任务，再交给多个 Sub-Agent 执行。

这时候断点续跑会更复杂。

每个子任务都可能处在不同状态：

- `PENDING`：还没开始。
- `RUNNING`：正在执行。
- `COMPLETED`：已经完成。
- `FAILED`：执行失败。
- `CANCELLED`：被取消。
- `TIMED_OUT`：超时了。

这些状态看起来像细节，其实是恢复逻辑的基础。

系统恢复的时候，不能粗暴地说“全部重新跑”。

它应该判断：

已经完成的子任务，不要重复执行。

失败的子任务，可以按策略重试。

超时的子任务，要先确认外部副作用。

被取消的子任务，要看取消动作是不是用户主动触发。

还没开始的子任务，可以继续排队。

你能讲清楚这一层，面试官基本能判断：你不是只玩过 Agent Demo，而是真的理解工程化执行。

## 一个更像样的面试回答

如果面试官问你：Agent 长任务中断以后，怎么实现断点续跑？

可以按这个结构回答：

第一，任务启动时生成唯一 run id，并初始化任务状态源。

第二，所有关键执行事实都写入统一状态对象，而不是只存在聊天记录里。

第三，在关键节点写 checkpoint，比如工具调用成功、外部副作用发生、用户确认、子任务完成。

第四，工具调用要设计幂等能力，至少要有请求 ID、业务唯一键或结果查询接口。

第五，系统恢复时先加载状态，再判断每个步骤是 completed、failed、running、timed out 还是 unknown。

第六，对于状态未知的动作，先做外部对账，再决定重试或跳过。

第七，恢复不是让模型自由发挥，而是由 Harness 根据状态驱动下一步执行。

这一套讲出来，就比“保存聊天记录再继续”高一个层级。

## 项目里怎么证明你真的做过

如果你在简历里写了 Agent 平台、AI 工作流、自动化执行系统，最好不要只写“接入大模型”“支持工具调用”。

这些已经不够有区分度。

更有说服力的是写：

- 设计了长任务状态模型。
- 实现了 checkpoint 与 resume。
- 支持工具调用结果持久化。
- 对外部副作用做了幂等处理。
- 支持失败重试、超时恢复和人工介入。
- 做过进程中断后的恢复测试。

最简单的测试也很直接：

让 Agent 执行一个 5 步任务。

第 3 步手动杀掉进程。

重启以后，看系统能不能识别前两步已经完成，第三步是否产生副作用，然后从正确位置继续。

这个测试，比你展示十次顺利运行的 Demo 更有价值。

因为真实系统不是看它顺风局多漂亮。

而是看它出事以后能不能收得住。

## 这就是 Agent 工程师的分水岭

未来一段时间，Agent 相关岗位会越来越多。

但面试和项目评估，也会越来越不满足于“我会调模型”。

真正拉开差距的，会是这些问题：

任务状态怎么设计？

工具调用怎么恢复？

外部副作用怎么去重？

子任务怎么管理生命周期？

失败之后怎么继续，而不是全部重来？

这些问题听起来没有模型参数那么性感。

但它们决定了 Agent 能不能真的进入生产环境。

所以，下一次有人问你 Agent 断点续跑，不要只说保存聊天记录。

你要让对方听出来：

你关心的不是一次回答，而是一套能在真实世界持续执行的系统。
