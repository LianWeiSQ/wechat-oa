# 面试官问：“你了解 Harness 吗？”我刚答完“就是给 Agent 加工具”，他下一句我就沉默了

## 那个让人立刻清醒的追问

面试快结束时，面试官看了眼我写的项目经历：

“你这里写了 Agent 自动调工具、自动执行任务。那我问个简单的，Harness 你怎么做的？”

我说：“就是给 Agent 接模型、接工具、接提示词。”

他停了一秒，继续追问：**“如果任务跑了 40 分钟挂掉，状态丢了、工具已经调了一半、其中一个写操作还可能成功了，你怎么让它继续跑，而不是重新闯祸？”**

这时候你会发现，气氛一下就变了。

因为这已经不是“你会不会用 Agent 框架”，而是在问：你有没有真正碰过 Agent 落地。

## 面试官问 Harness，真的不是想听一个定义

最近很多人把 Harness 当成一个新术语来背。

但面试官真正在测的，往往不是你知不知道这个词，而是你有没有意识到一件事：

**Agent 从 Demo 到生产，中间隔着的不是模型能力，而是一整套运行托底系统。**

Prompt 解决的是“怎么说”。

Context 解决的是“给它什么信息”。

而 Harness 解决的是“它开始连续行动之后，怎么别跑偏、别越权、别失忆、别重复犯错”。

说白一点：

- Demo 里的 Agent，重点是“能做出一次结果”
- 生产里的 Agent，重点是“多次运行都可控、可恢复、可追责”

所以，面试官一问 Harness，背后其实在问三层东西：

1. **执行层**：Agent 到底在什么环境里跑，怎么调工具，怎么编排长任务
2. **约束层**：它能看什么、能做什么、哪些动作必须拦住
3. **状态层**：它做到哪了，挂了怎么恢复，失败了怎么重试，做错了怎么回滚

这才是 Harness 的分量。

## 一句话先答对：什么是 Harness

如果面试时只能先说一句，我建议这么答：

**Harness 是围绕模型外侧搭建的一层运行系统，用来把 Agent 的执行环境、工具权限、上下文规则、状态恢复和观测机制工程化，从而让它能稳定完成长链路任务。**

注意，这句话有两个关键词。

一个是“模型外侧”。

意思是：Harness 不是模型本身，而是模型之外那一整圈工程系统。

另一个是“稳定完成长链路任务”。

意思是：Harness 不是为了让一次演示更惊艳，而是为了让十次、百次任务都不失控。

## 反过来看：为什么很多 Agent 项目一上生产就露馅

因为很多团队做的是“会调用工具的聊天机器人”，不是“可运行的 Agent 系统”。

表面上它已经能：

- 读需求
- 规划步骤
- 调接口
- 返回结果

但只要任务一变长，问题就全来了。

比如：

- 上下文越堆越多，前面定的规则被冲掉了
- 工具调失败了，只会机械重试
- 写操作超时了，不知道到底成功没有
- 跑到一半进程挂了，下次只能从头再来
- 删除、退款、发券这类高风险动作没有人工确认
- 线上出了问题，根本查不到当时模型为什么这么决策

这时候你会意识到，Agent 最难的不是“会思考”，而是“怎么被系统托住”。

这也是我们更关心模型背后的工程栈，而不是只看谁又发了一个更会聊天的新模型。模型会变，工程问题不会自动消失。

## 一个可复用的面试回答框架

下面这套框架，基本可以直接复用。

先给定义，再讲五个关键模块，最后接一个项目案例。面试官会明显感觉你不是在背词，而是在讲系统。

## 模块一：执行环境，Agent 到底在哪里跑

先回答运行时。

**执行环境**决定 Agent 能访问什么资源、以什么身份运行、任务如何被调度。

这里最少要讲三件事：

- 是本地沙箱、容器、远程工作空间，还是业务后端宿主
- 工具调用是在同进程内执行，还是通过 MCP / API / Job System 远程执行
- 长任务是同步阻塞跑，还是异步入队，由 worker 持续推进

为什么这层重要？

因为一旦 Agent 真要干活，它就不再只是吐文本，而是在操作文件、服务、数据库、工单系统、代码仓库。没有清晰的执行边界，安全性和稳定性都谈不上。

一句好用的回答是：

**我会先把 Agent 放进受控执行环境里，隔离文件系统、网络和身份，再通过统一工具网关暴露能力，而不是让模型直接碰真实系统。**

## 模块二：上下文、记忆、项目规则，不是让模型看到更多

这是很多人最容易讲浅的地方。

面试里别只说 RAG，太窄了。

Harness 里的上下文管理，至少分三类：

1. **当前任务上下文**：这次用户要什么、目标是什么、约束是什么
2. **项目长期规则**：项目结构、命令、禁改范围、验证方式，比如类似 `CLAUDE.md` 这种项目级规则
3. **运行中记忆**：阶段性结论、已经完成的步骤、待办清单、handoff 文档

关键不是“把所有资料都塞进去”，而是**分层、按需、可沉淀**。

因为临时对话里的规则会被长上下文稀释，真正稳定的规则必须外化成项目记忆；而长任务中的中间状态，也不能只存在模型脑子里，必须落成结构化 artifact。

面试时你可以这样说：

**我会把跨任务稳定规则沉淀成项目级记忆，把本次任务目标放当前 prompt，把中间产物写成可交接的状态文件，这样上下文压缩或会话切换时，Agent 仍然能接着干。**

## 模块三：工具权限，不是“能调多少工具”

面试官只要问 Harness，大概率会往工具安全追问。

这里一定要主动区分工具类型：

- **只读工具**：查订单、查日志、搜知识库、读代码
- **写入工具**：发券、退款、改配置、删文件、提 PR、发邮件

这两类工具的治理方式完全不一样。

只读工具可以更自动化。

写入工具必须加约束：

- 参数校验
- 身份与权限校验
- 风险分级
- 高风险动作人工确认
- 不可逆操作默认禁止自动执行

如果你只说“让模型根据 tool schema 调用”，面试官基本知道你还停留在 function calling 演示阶段。

更像生产回答的说法是：

**工具调用不是模型直接执行命令，而是模型提出意图，执行层先做权限、参数、风险校验，再决定是否放行。**

## 模块四：状态与 checkpoint，长任务不能靠模型“记得”

这是 Harness 最容易把人区分开的部分。

一个成熟的 Agent，必须有明确状态流转，比如：

- PENDING
- RUNNING
- WAITING_TOOL
- WAITING_HUMAN
- CHECKPOINTED
- FAILED
- COMPLETED

更重要的是：**每个关键步骤后要不要落 checkpoint。**

什么叫 checkpoint？

就是把“当前做到哪一步、已经产出什么、下一步该干什么、哪些动作已经提交过”持久化下来。

这样做有三个好处：

1. 进程挂了可以恢复
2. 长任务可以断点续跑
3. 可以做 context reset，让新会话接手，而不是让同一个上下文无限膨胀

很多长任务不是因为模型不会做，而是做着做着失焦。这个时候，重开一个干净上下文，让它从结构化 handoff 接着跑，往往比硬压缩上下文更稳。

所以你可以明确讲：

**我不会让长任务状态只停留在对话历史里，而是把计划、已完成步骤、工具回执和待确认事项落到 checkpoint，支持恢复和会话切换。**

## 模块五：失败恢复、幂等、回滚、人工确认与可观测性

这一块，是面试里最见真章的地方。

因为现实世界里，失败不是例外，是常态。

这里建议你按顺序讲。

### 1. 失败先分类，不是先重试

- 参数错误：不重试，应该回到规划层修正
- 权限错误：不重试，应该拒绝或申请授权
- 业务规则错误：不重试，比如“已发货订单不可取消”
- 网络抖动或瞬时故障：可以有限次重试
- 写操作超时：先查状态，不能直接重试，因为结果可能未知

### 2. 写操作必须幂等

发券、退款、创建工单、提交审批、发邮件，这些都要带幂等键。

同一个任务步骤重复提交，业务系统应该识别为同一请求，而不是再执行一次。

### 3. 可撤销动作要有回滚或补偿

比如：

- 创建草稿可以删除草稿
- 工单转派可以撤回转派
- 配置变更可以恢复旧版本

不是所有动作都能真回滚，但至少要有补偿设计。

### 4. 高风险动作要人工确认

尤其是扣款、删除、外发邮件、线上配置修改。

Agent 可以准备方案、填好参数、展示影响范围，但最后一步应等待确认。

### 5. 全链路可观测

每次任务都要能追：

- 会话 ID / 任务 ID / step ID
- 模型输入摘要
- 工具调用参数摘要
- 错误码、重试次数、耗时
- checkpoint 位置
- 人工确认人和时间

没有 trace，你就只能靠猜。

所以这一段可以浓缩成一句：

**生产级 Harness 的关键，不是让 Agent 永远不出错，而是出错后能识别、能阻断、能恢复、能审计。**

## 把五个模块串起来，给面试官一个全景图

你可以直接这样画：

- **执行环境**：Agent 在哪里跑，如何调度
- **上下文与记忆**：这轮看什么，长期规则放哪里
- **工具权限**：哪些能力可用，哪些动作要拦
- **状态与 checkpoint**：做到哪一步，如何恢复
- **失败恢复与观测**：怎么重试、幂等、回滚、确认、追踪

如果想再上一个层次，可以补一句：

**这五层其实对应三件事：让 Agent 能执行、让 Agent 不越界、让 Agent 挂了还能爬起来。**

这就是面试官真正想听到的结构感。

## 一个项目案例式回答，建议直接背到自己的语言里

假设你做的是“售后运营 Agent”，要自动查询订单、判断可退性、生成补偿方案、必要时发券或创建工单。

你可以这么答：

**一句话项目定义**：
我们做的不是一个会聊天的客服助手，而是一个带 Harness 的业务 Agent，目标是让它在受控条件下完成跨系统售后处理。

**执行环境**：
Agent 运行在后端 worker，不直接暴露数据库权限；所有业务能力都通过工具网关访问，包括查询订单、查询物流、创建工单、发券。长任务采用异步 job 执行，避免前端请求超时。

**上下文与规则**：
当前任务目标由用户请求和工单内容组成；项目级规则单独维护，比如补偿上限、禁止直接退款的场景、各类售后 SOP；运行中会把“已确认的用户意图、已查询到的订单状态、候选处理方案”写入状态对象，而不是只放在聊天历史里。

**工具权限**：
查询类工具默认可自动调用；发券、退款、关闭工单这类写工具要先过权限和参数校验。高风险动作需要人工确认，像“给用户补发 100 元券”必须展示理由、金额、影响范围，再等待运营点击确认。

**状态与 checkpoint**：
任务会按步骤落 checkpoint，比如“已完成订单查询”“已完成规则判断”“待人工确认”“已提交工单”。如果 worker 重启或超时，可以从上一个 checkpoint 续跑；如果上下文太长，也可以由新会话根据 checkpoint 和 handoff 文档接手。

**幂等、重试与回滚**：
查询类接口超时可以自动重试；发券这类写操作必须带幂等键，超时后先查业务状态，不能直接二次提交。可撤销动作设计补偿工具，比如错误创建的工单可以自动关闭；不可逆动作不允许 Agent 全自动执行。

**可观测性**：
我们会记录每次任务的 trace，包括模型决策摘要、工具调用链路、错误码、重试行为、人工确认节点和最终结果。这样线上一旦出现重复发券或错误判责，可以复盘到具体 step。

最后收一句：

**所以我理解 Harness，不是多接几个工具，而是把 Agent 的执行层、约束层、状态层补齐，保证它能安全地把任务跑完。**

这段话一出来，面试官通常就知道你理解的不是玩具系统。

## 最后一句话：别把 Harness 当新瓶装旧酒，也别把它神化

Harness 不是一个花哨概念，更不是某个框架专属黑话。

它本质上是在提醒我们：

**模型能力再强，真正决定 Agent 能不能进生产的，仍然是它背后的工程栈。**

所以以后再被问到 Harness，不要急着从名词解释开始。

先想三个问题：

- 它在什么环境里执行？
- 它被什么规则约束？
- 它出问题后怎么恢复？

能把这三件事讲清楚，你回答的就不只是 Harness。

你回答的是：你到底懂不懂，Agent 为什么总是卡在从 Demo 走向生产的那一步。
