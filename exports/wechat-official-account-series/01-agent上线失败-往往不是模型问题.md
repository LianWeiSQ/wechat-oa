# Agent 上线失败，往往不是模型问题

摘要：

很多 Agent 项目在 Demo 阶段表现很好，一到真实生产环境就开始出现长任务中断、工具调用重复、状态丢失和上下文续不上。问题不一定在模型，而在负责状态、工具、执行和恢复的 Harness。

封面 Brief：

技术杂志风封面。画面中心是一套 Agent 运行管线：模型、工具、状态存储、恢复闭环和用户任务。风格克制、清晰、有工程感，不要密集文字，不要卡通机器人。

## 不是模型不聪明，是系统接不住

很多团队第一次做 Agent，最容易把注意力放在模型上。

要不要换更强的模型？要不要上更长上下文？要不要同时接几个模型做路由？

这些问题当然重要，但它们经常掩盖一个更底层的问题：模型生成的计划，最后要由谁稳定地执行？

我见过不少项目，测试环境里表现很好。用户提一个任务，Agent 拆成几个步骤，调用工具，输出结果，看起来已经接近可用了。可一进入真实环境，问题马上变得具体：工具接口超时，用户中途刷新，网络抖动，某个子任务卡住，第三方 API 返回了一个模糊错误。

这时候模型没有变，Prompt 也没有变，系统却开始不稳定。

根因通常不是模型不会思考，而是模型之外的执行系统没有准备好。

这层系统，我会统一称为 Harness。

## Harness 是什么

不要把 Harness 理解成一个时髦词。它不是简单的调度器，也不是把模型和工具粘在一起的胶水代码。

在 Agent 系统里，Harness 至少要管五件事：

第一，入口适配。用户可能来自网页、API、命令行、企业微信、飞书或自动任务。不同入口的上下文、权限和生命周期都不一样，Harness 要把它们收敛成统一的执行模型。

第二，会话成型。一次 Agent 执行不是一句问答，而是一组可追踪的 turn。每个 turn 有输入、有中间动作、有工具结果、有状态更新，也要有失败时的恢复点。

第三，状态管理。Agent 执行过程中会产生大量中间状态：当前执行到哪一步、哪些工具已经调用、哪些结果已经返回、哪些动作已经产生外部副作用。这些状态必须被组织成一个可信结构。

第四，工具编排。工具不是函数调用那么简单。工具有权限、超时、重试、幂等、副作用和降级策略。Harness 必须知道什么时候能重试，什么时候不能重试。

第五，异常恢复。真实系统一定会失败。区别只在于失败后能不能恢复，能不能告诉用户发生了什么，能不能从上一个可靠状态继续。

换句话说，模型负责判断下一步可能应该做什么，Harness 负责保证这一步能被可靠执行。

## 为什么 Demo 很强，上线很弱

Demo 阶段的问题通常是短链路问题。

用户给一个任务，模型调用一两个工具，系统返回一个结果。只要模型能力够强，效果就很容易让人兴奋。

但生产环境的问题是长链路问题。

一个真实任务可能持续十几轮甚至几十轮，中间穿插多个工具、多个状态对象、多个外部系统。只要其中一个环节没有被正确记录，后面就会开始失真。

比如 Agent 负责帮销售整理客户资料，并自动写入 CRM。

如果写入 CRM 的动作成功了，但 Harness 只记录了“调用过 CRM 工具”，没有记录外部系统返回的客户 ID，那么任务中断后再恢复，系统就不知道这个客户到底有没有创建成功。

这时有两个坏选择：

要么重新创建，可能产生重复客户。

要么跳过这一步，可能漏掉关键数据。

这不是模型能靠更聪明解决的问题。因为问题发生在外部世界，发生在系统状态和真实副作用之间。

## 一个判断：模型决定上限，Harness 决定下限

更强的模型会提升 Agent 的上限。它能更好地理解任务，更准确地拆步骤，更少产生无效工具调用。

但系统的下限由 Harness 决定。

如果状态管理混乱，模型再强也会在长任务中丢上下文。

如果工具调用没有幂等设计，模型再强也可能重复执行外部动作。

如果没有断点续跑，模型再强也救不了进程崩溃后的任务。

如果异常没有分级处理，模型再强也只能在错误发生后生成一段看似合理的解释。

这就是为什么很多 Agent 项目看起来是模型问题，最后查出来都是工程问题。

## 上线前先问五个问题

如果你正在做 Agent 项目，可以先不讨论模型，先问团队五个问题。

第一，Agent 的唯一可信状态源在哪里？

第二，每次工具调用的请求、结果和副作用是否被结构化记录？

第三，任务中断后，系统能不能从上一个可靠状态恢复？

第四，外部动作是否有幂等 ID，避免重复执行？

第五，失败状态是否有细分，还是只有成功和失败两个结果？

这五个问题里，如果有两个回答不清楚，系统大概率还停留在 Demo 阶段。

## 真正的工程分水岭

Agent 工程化的分水岭，不是第一次让模型调用工具。

而是第一次跑通这条链路：

任务开始，执行几步，中途故意中断，系统重启，读取状态，识别已完成动作，跳过不可重复步骤，继续执行，最后给用户一个完整结果。

这条链路跑通，说明你的 Agent 开始具备生产系统的基本形态。

跑不通，说明你拥有的是一个会调用工具的聊天系统，还不是一个可靠的 Agent 系统。

## 结尾

今天很多人谈 Agent，喜欢谈模型、谈 Prompt、谈工具生态。

这些都值得谈。

但如果目标是上线，而不是做一个演示，问题最终会回到 Harness：状态怎么存，工具怎么管，失败怎么恢复，副作用怎么追踪。

模型能让 Agent 看起来更聪明。

Harness 才能让 Agent 真的可用。
