# Agent 长任务为什么需要唯一状态源

摘要：

短会话 Agent 可以靠模型记忆和临时变量撑住，长任务不行。长任务真正需要的是一个唯一可信状态源，持续记录当前步骤、工具结果、外部副作用和恢复位置。

## 长任务不是多聊几轮那么简单

很多人把 Agent 长任务理解成“上下文更长的聊天”。

这个理解会让系统设计走偏。

长任务当然需要上下文，但上下文不是全部。一个真实长任务里，系统不仅要知道用户说过什么，还要知道任务执行到哪里、哪些工具已经调用、哪些结果已经确认、哪些外部动作已经产生副作用、哪些步骤可以重试。

这些东西不能只放在模型上下文里。

模型上下文适合承载语义信息，不适合承担系统状态的唯一真相。

如果把状态都塞进上下文，短期看实现很快，长期看一定会失控。因为上下文会被压缩、裁剪、总结，也可能在多轮对话中逐渐变形。

Agent 的状态，必须有一份模型之外的可信载体。

## 什么是唯一状态源

唯一状态源可以叫 ThreadState，也可以叫 AppState，名字不重要。

重要的是：一次 Agent 执行的关键事实，最终都要汇聚到同一个状态对象里。

它至少应该记录：

- 用户目标是什么。
- 当前执行到哪一步。
- 已完成的子任务有哪些。
- 每次工具调用的请求参数和返回结果是什么。
- 哪些外部动作已经提交成功。
- 哪些步骤失败了，失败原因是什么。
- 当前任务能不能继续，继续要从哪里开始。

这份状态不是日志。

日志记录发生过什么，方便排查。

状态源记录系统现在相信什么，决定下一步怎么执行。

这是两件不同的事。

## 没有唯一状态源，会发生什么

最常见的问题是状态散落。

一部分状态在前端，一部分状态在后端内存，一部分状态在数据库，一部分状态在模型上下文，一部分状态藏在工具返回里。

系统正常运行时，看起来没有问题。因为所有东西都还在内存里，调用链也没有断。

一旦进程重启、请求超时、用户刷新页面、队列任务被重新消费，问题就出现了。

没有一个地方能回答：

这个任务现在到底完成到哪一步？

这不是小问题。它会直接影响恢复、重试、排错和用户信任。

用户看到的是“任务失败了”。

工程团队看到的是“一堆状态对不上”。

管理者看到的是“Agent 项目不稳定”。

最后很容易误判成模型能力不够。

## 状态源要和模型上下文分工

模型上下文应该负责理解和推理。

状态源应该负责事实和执行。

比如用户让 Agent 分析 20 个客户，并给每个客户生成一段跟进建议。

模型上下文可以放客户背景、任务要求、写作偏好。

状态源必须记录 20 个客户分别处理到哪一步：已读取资料、已生成建议、已写入 CRM、写入失败、等待人工确认。

如果只把这些信息塞进模型上下文，后面很难做可靠恢复。

因为模型可能在总结历史时漏掉一个客户，也可能把“待确认”说成“已完成”。对生成文本来说，这只是语义误差；对生产系统来说，这就是状态事故。

## 状态源不是越大越好

唯一状态源不等于把所有东西都塞进去。

状态设计要克制。

一份好的 Agent 状态，应该保留决策所需的关键事实，而不是保存所有原始内容。

可以把状态分成三层：

第一层是任务级状态：任务目标、用户、权限、整体进度、最终交付物。

第二层是步骤级状态：每个子任务的状态、输入、输出、错误和恢复策略。

第三层是工具级状态：工具请求、外部结果 ID、幂等 ID、提交状态和重试次数。

原始大文件、长文本、完整工具响应，可以放到对象存储或日志系统中，状态源只保存引用和关键摘要。

这样既能恢复，又不会让状态对象变成不可维护的垃圾桶。

## 什么时候持久化

状态只在内存里是不够的。

但每一步都全量写数据库，也可能带来性能问题。

实践中可以区分关键节点和普通节点。

普通节点可以异步记录，比如模型中间思考、临时草稿、可重复计算的内容。

关键节点必须同步持久化，比如外部工具提交成功、子任务完成、用户确认、不可逆动作执行前后。

一个简单原则是：凡是恢复后不能靠重新计算安全得到的事实，都应该及时持久化。

比如“邮件内容草稿”可以重新生成。

但“邮件已经发送给谁，外部 messageId 是什么”必须持久化。

## 一个最小实现建议

不要一开始就做复杂状态平台。

最小可用版本可以很朴素：

一个任务表，记录任务整体状态。

一个步骤表，记录每个子任务的状态流转。

一个工具调用表，记录请求、响应、幂等 ID、外部结果 ID。

再加一个恢复函数：读取任务，找到最后一个可靠步骤，判断下一步是继续、重试、跳过还是人工介入。

这套东西不华丽，但它比只靠上下文可靠得多。

等业务复杂起来，再考虑事件溯源、工作流引擎、分布式锁、队列和状态机框架。

不要反过来，一开始就堆框架，却没有想清楚状态事实是什么。

## 结尾

Agent 长任务的本质，不是更长的聊天。

它是一条持续变化的执行链。

执行链要可靠，系统就必须有一份可信状态源，清楚记录已经发生的事实、当前所处的位置和下一步能做什么。

模型可以负责推理。

但状态真相不能交给模型记忆。
