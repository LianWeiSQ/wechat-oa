# 工具调用成功了，Agent 为什么还是会出事故

摘要：

Agent 接工具很容易，管好工具很难。真正的生产风险通常出现在工具调用之后：外部动作已经发生，但系统状态没有记录清楚，恢复或重试时就可能重复执行、漏执行或错误继续。

封面 Brief：

画面是一个工具调用链路图：Agent 发起请求，外部系统返回结果，Harness 写入状态源。突出“请求 ID、结果 ID、提交状态、重试策略”四个标签。风格清晰专业。

## 接上工具，只是开始

很多 Agent 项目做到工具调用那一步，会有一种明显的兴奋感。

模型不再只是聊天，它可以查数据库、发邮件、建工单、写文档、调用内部系统。

这当然是 Agent 从“回答问题”走向“执行任务”的关键一步。

但也正是从这一步开始，系统风险变得真实。

因为工具调用会改变外部世界。

聊天机器人说错一句话，最多重新回答。

Agent 发错一封邮件、重复创建一个订单、覆盖一条客户记录，后果就不是重新生成文本那么简单。

所以评估 Agent 工具层，不要只问“能不能调用”。

更重要的问题是：调用成功以后，系统有没有把结果接住？

## 最危险的状态：工具成功，系统不知道

考虑一个很常见的场景。

Agent 帮运营同事创建活动页，流程包括生成文案、上传图片、创建页面、发布链接、通知相关人员。

创建页面这一步调用 CMS API 成功了，外部系统返回了页面 ID。

但 Harness 没有把这个页面 ID 写入状态，只是在日志里留下了一行“create page success”。

接下来任务中断。

恢复后，系统读不到页面 ID，只知道之前执行到创建页面附近。

这时候怎么办？

重新创建页面，可能出现重复页面。

跳过创建页面，后面的发布链接又没有对象。

让模型判断，它也只能根据残缺上下文猜。

事故就这样发生了。

这个问题看起来像异常处理，实质是副作用管理。

外部动作一旦成功，Harness 必须把它变成系统状态里的可靠事实。

## 工具调用至少要记录什么

一个能上线的 Agent 工具调用，不应该只记录“调用了哪个工具”。

至少要记录五类信息。

第一，请求 ID。

每次工具调用都要有内部请求 ID，方便追踪这次调用从哪里来，属于哪个任务、哪个步骤。

第二，幂等 ID。

对于可能产生副作用的动作，必须有幂等 ID。重试时外部系统能识别这是同一次业务动作，而不是新的动作。

第三，外部结果 ID。

比如 messageId、orderId、ticketId、pageId、recordId。这是外部世界已经发生变化的证据。

第四，提交状态。

要区分请求已发送、外部已接收、外部已提交成功、外部返回失败、状态未知。状态未知是最麻烦的一类，不能简单当成失败。

第五，恢复策略。

恢复时这一工具调用应该跳过、重查、重试，还是人工确认。

没有这些记录，工具越多，风险越大。

## 为什么不能简单重试

很多后端系统都强调重试。

但 Agent 工具层不能把重试当万能药。

因为有些工具调用是可重复的，比如查询天气、读取文档、搜索资料。

有些工具调用是不可随便重复的，比如发邮件、扣款、下单、改权限、写入 CRM。

查询失败可以重试。

扣款状态未知，不能直接再扣一次。

这就是幂等设计的价值。

幂等不是为了优雅，而是为了让系统在不确定状态下仍然有安全边界。

Agent 工具调用的工程原则很简单：

能重试的动作，要记录重试次数和退避策略。

不能重复的动作，要有幂等 ID 和外部结果确认。

状态未知的动作，要先查外部系统确认，而不是让模型猜。

## Observe 阶段必须结构化

很多 Agent 设计会讲 Think-Act-Observe-Repeat。

但真正决定稳定性的，经常是 Observe。

Act 是调用工具。

Observe 是把工具结果变成系统能理解的事实。

如果 Observe 只是把原始返回文本丢给模型，那 Harness 并没有真正掌握执行结果。

正确做法是把工具返回结构化：

哪些字段是业务结果？

哪些字段是外部结果 ID？

哪些字段代表可恢复错误？

哪些字段代表不可重试错误？

哪些字段需要写入状态源？

这一步做好了，后面的恢复、审计、用户解释都容易很多。

这一步偷懒，系统只能靠模型上下文延续，迟早会在长任务里出问题。

## 工具层要有权限和降级

除了幂等，工具层还有两个容易被低估的问题：权限和降级。

Agent 不应该天然拥有所有工具权限。

不同用户、不同任务、不同环境，应该装配不同的工具能力。能读不一定能写，能生成不一定能发布，能创建草稿不一定能直接发送。

这不是形式主义。

Agent 一旦误判，权限边界就是最后一道保护。

降级也一样。

工具调用失败时，不是只有“报错退出”一种选择。

有些失败可以换备用数据源。

有些失败可以先生成草稿，等待人工补充。

有些失败必须停止，并明确告诉用户哪个外部系统不可用。

这些策略不能临时交给模型发挥，需要 Harness 预先定义。

## 一个判断标准

判断一个 Agent 工具层是否靠谱，可以看一个问题：

如果工具调用成功后，系统立刻崩溃，重启后能不能准确知道外部世界发生了什么？

如果答案是不知道，那工具层还没有生产化。

工具调用不是函数调用。

它是 Agent 和真实世界之间的边界。

边界越真实，状态越要严谨。

## 结尾

Agent 的能力感来自工具。

但 Agent 的可靠性来自 Harness 对工具副作用的管理。

能接工具，只说明系统开始具备行动能力。

能记录工具结果、处理幂等、识别状态未知、支持恢复和降级，才说明系统开始具备生产能力。
