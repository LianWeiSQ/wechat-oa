# 断点续跑：Agent 从 Demo 到生产的分水岭

摘要：

Agent 系统一定会遇到中断。网络会抖，工具会超时，进程会重启，用户会离开页面。断点续跑不是锦上添花，而是长任务 Agent 进入生产环境的基本能力。

封面 Brief：

画一条长任务执行时间线，中间出现中断点，系统从最近一次可靠快照恢复继续。标出 checkpoint、resume、tool result、state store。风格像技术架构图。

## 失败不是例外，是常态

很多 Agent Demo 默认一个前提：任务会顺利执行完。

用户输入，模型规划，工具调用，结果返回。

但生产环境不会这么配合。

真实用户可能在任务跑到一半时关掉页面。内部 API 可能偶发 502。模型调用可能超时。队列 worker 可能重启。外部系统可能返回一个状态未知的结果。

如果每次中断都只能让用户重新开始，这个 Agent 很难承担真实工作。

断点续跑的价值，不是让系统显得高级。

它是长任务 Agent 的基本生存能力。

## 断点续跑到底续的是什么

很多人一听断点续跑，以为是保存聊天记录。

不够。

聊天记录只能告诉你对话说到哪里，不能告诉你任务执行到哪里。

断点续跑真正要恢复的是执行状态。

包括：

- 当前任务目标。
- 已完成步骤。
- 正在执行但未确认的步骤。
- 已产生副作用的工具调用。
- 可安全重试的失败动作。
- 不可重复执行的外部动作。
- 下一步应该继续、跳过、重试还是等待人工确认。

如果这些信息没有结构化保存，恢复只是表面恢复。

系统看起来重新打开了任务，实际上并不知道该从哪里继续。

## 最小可行的恢复闭环

一个最小可行的断点续跑闭环，可以分成五步。

第一，执行前建立任务状态。

任务一开始，系统就要创建任务 ID，并初始化状态源。不要等执行到一半才想起来记录。

第二，每个关键步骤完成后写入 checkpoint。

不是所有中间过程都要持久化，但关键事实必须落盘。尤其是工具调用成功、用户确认、外部动作提交这些节点。

第三，中断后重新加载状态。

恢复时不要只加载聊天历史，而要加载任务状态、步骤状态和工具调用记录。

第四，识别未完成动作。

系统要判断上次中断时，哪些动作已经完成，哪些动作状态未知，哪些动作可以重试。

第五，从可靠位置继续。

继续不是简单从最后一行日志往下跑，而是基于状态判断下一步。

这五步跑通，系统才真正具备断点续跑能力。

## 状态未知是最难处理的

断点续跑里最麻烦的不是成功，也不是失败。

是状态未知。

比如 Agent 调用外部系统创建工单，请求发出去了，但响应没回来。网络在这时断了。

外部系统到底创建成功了吗？

不知道。

如果直接重试，可能重复创建。

如果直接跳过，可能漏掉工单。

这时 Harness 需要一个“状态确认”步骤：用幂等 ID 或业务查询条件去外部系统查，确认这个动作是否已经发生。

所以断点续跑不是只靠本地 checkpoint。

它还要求工具层支持结果查询、幂等识别和状态对账。

没有这些能力，恢复逻辑会被迫猜测。

生产系统最怕猜。

## 为什么断点不能太粗

有些团队为了省事，只在任务开始和任务结束保存状态。

这对短任务可以，对长任务不够。

假设一个 Agent 要处理 100 条客户记录，只在最后写一次结果。如果处理到第 98 条时失败，系统要么重头处理，要么完全不知道哪些客户已经完成。

更合理的方式是按可恢复单元保存。

比如每处理完一个客户写一次步骤状态，或者每完成一个批次写一次 checkpoint。

断点粒度太粗，恢复成本高。

断点粒度太细，持久化开销大。

这就是 Harness 要做的权衡。

可以从简单规则开始：

凡是重跑成本高、会产生外部副作用、用户已经看见结果的节点，都应该成为 checkpoint。

## 给团队的一个测试

如果你想判断自己的 Agent 是否接近生产化，可以做一个很朴素的测试。

让 Agent 执行一个包含 5 个步骤的任务，其中至少 2 个步骤会调用外部工具。

执行到第 3 步时，手动杀掉进程。

然后重启系统，看它能不能：

读取原任务。

识别前 2 步已经完成。

确认第 3 步是否产生外部副作用。

从正确位置继续。

最后输出完整结果。

这个测试比一百次顺利 Demo 更有价值。

因为它测试的是系统在不完美环境中的恢复能力。

## 不要一开始追求完美

断点续跑可以做得很复杂。

事件溯源、分布式状态机、工作流引擎、补偿事务、任务编排平台，都可以进入讨论。

但早期不必一上来就堆这些。

先做最小闭环：

任务状态可查。

步骤状态可追踪。

工具调用有记录。

关键节点有 checkpoint。

恢复函数能决定继续、重试、跳过或人工介入。

只要这条线跑通，后面再逐步增强。

最糟糕的是反过来：框架很多，状态事实不清楚。

## 结尾

可靠的 Agent 不是不失败。

可靠的 Agent 是失败后不慌。

它知道哪些事情已经发生，哪些事情不能重复，哪些事情可以重试，哪些事情必须交给人。

断点续跑不是一个漂亮功能。

它是 Agent 从 Demo 走向生产的分水岭。
