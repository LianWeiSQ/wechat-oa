# CTO 如何判断 Agent 项目是不是在空转

摘要：

很多企业的 Agent 项目长期停在 Demo 和试点之间。判断项目是不是在空转，不要只看模型效果和演示视频，而要看任务边界、状态管理、工具权限、恢复能力和业务指标是否被真正设计出来。

封面 Brief：

管理视角的技术决策图。左侧是“Demo 热闹”：模型、Prompt、演示视频。右侧是“生产系统”：任务边界、状态源、工具权限、恢复、指标。风格像咨询报告封面，但保持技术感。

## Agent 项目最容易卡在中间

很多企业做 Agent，前两周进展很快。

接一个模型，写几个 Prompt，挂几个工具，做一个演示流程。

老板看了觉得有希望，业务同事觉得很新鲜，技术团队也能看到方向。

但再往后，项目容易进入一种尴尬状态：

Demo 一直在优化，真实业务却迟迟接不进去。

会议一直在讨论模型效果，没人说清楚上线边界。

每次试点都能跑通几个样例，但一遇到异常就需要工程师手动介入。

这就是 Agent 项目空转的典型状态。

表面上团队很忙，实际没有接近生产。

## 第一个信号：一直在换模型

如果一个 Agent 项目连续几周的主要进展都是换模型、调 Prompt、比较回答质量，就要警惕。

模型评估当然重要。

但如果项目目标是上线，模型只是其中一层。

真正该同时推进的问题包括：

这个 Agent 到底负责哪个业务任务？

任务完成的定义是什么？

哪些步骤可以自动执行，哪些必须人工确认？

工具权限怎么控制？

失败后谁接管？

业务指标怎么衡量？

如果这些问题没有答案，换模型只是在提高 Demo 的观感。

它不会自动把项目推向生产。

## 第二个信号：没有明确任务边界

“做一个销售 Agent”“做一个运营 Agent”“做一个客服 Agent”，这些说法都太大。

大到无法上线。

一个可以落地的 Agent 任务，边界应该更具体：

读取指定客户资料，生成跟进建议，写入 CRM 草稿，等待销售确认。

根据活动目标生成 3 版推文草稿，自动检查敏感词，提交给运营审核。

根据工单内容检索知识库，生成回复建议，低风险问题自动回复，高风险问题转人工。

边界越清楚，状态、工具、权限和指标才有设计基础。

没有任务边界，Harness 无从设计。

最后系统只能变成一个什么都能聊、什么都不敢真正执行的助手。

## 第三个信号：没人负责状态

CTO 可以问团队一个很具体的问题：

一次 Agent 任务跑到一半失败后，我们怎么知道它完成到哪一步？

如果答案是“看日志”或者“让模型总结上下文”，说明还没有进入生产系统设计。

状态管理应该是 Agent 项目的核心工程资产。

它决定任务能不能恢复，错误能不能定位，用户能不能看到进度，业务能不能审计。

一个认真推进的 Agent 项目，应该至少设计三类状态：

任务状态：整体目标、进度、负责人、结果。

步骤状态：每个子任务的完成情况、错误和恢复策略。

工具状态：每次外部调用的请求、结果、幂等 ID 和副作用。

如果这些都没有，项目还停留在“模型应用”阶段，不是“Agent 系统”阶段。

## 第四个信号：工具权限过大

很多 Demo 为了方便，会给 Agent 开很大的工具权限。

能读、能写、能发、能删、能发布。

这在演示时很爽，在生产里很危险。

企业里的 Agent 不应该天然拥有完整权限。

它应该根据任务、用户、环境和风险等级装配工具。

比如客服 Agent 可以读取订单和知识库，但退款必须人工确认。

销售 Agent 可以写 CRM 草稿，但不能直接覆盖关键字段。

运营 Agent 可以生成活动页草稿，但发布必须经过审批。

权限边界不是降低效率，而是让自动化可控。

没有权限设计的 Agent，一旦出错，很难解释责任边界。

## 第五个信号：没有恢复演练

CTO 不需要亲自看每一行代码，但可以要求团队做一个恢复演练。

选一个真实任务，执行到一半，故意中断。

然后观察系统能否恢复：

能不能找到原任务？

能不能识别已完成步骤？

能不能确认外部工具是否已经成功？

能不能避免重复执行不可逆动作？

能不能给用户一个清楚的进度说明？

这个演练非常有效。

因为它会立刻暴露项目是 Demo 逻辑，还是生产逻辑。

Demo 追求顺利路径。

生产系统必须处理不顺利路径。

## 第六个信号：指标只有“效果不错”

Agent 项目不能只用“回答质量不错”“业务同学觉得可以”来判断进展。

需要把指标拆开。

效率指标：节省了多少人工时间？

质量指标：结果准确率、人工修改率、拒绝率是多少？

稳定性指标：任务成功率、恢复成功率、工具失败率是多少？

成本指标：单任务模型成本、工具成本、人工审核成本是多少？

风险指标：错误执行、重复执行、越权操作出现过几次？

这些指标不一定一开始就完美，但必须进入项目视野。

否则团队会一直围绕主观体验打转。

## 一条判断线

判断 Agent 项目是不是在空转，可以看会议里讨论的问题。

如果大家主要讨论：

模型哪个好；
Prompt 怎么写；
Demo 怎么更惊艳；
回答怎么更像人。

项目可能还在早期探索。

如果大家开始讨论：

任务边界；
状态源；
工具幂等；
权限控制；
断点恢复；
人工接管；
业务指标。

项目才开始进入真实工程阶段。

这不是说前者不重要。

而是只有进入后者，Agent 才有机会从演示走向业务系统。

## 给 CTO 的推进建议

不要一上来要求团队做一个万能 Agent。

选一个窄任务。

定义清楚输入、输出、工具、权限、人工确认点和失败接管方式。

先跑通一条可恢复的执行链路。

然后再扩任务范围、扩工具能力、扩自动化比例。

Agent 项目最怕一开始愿景很大，工程闭环很薄。

真正有效的路线通常相反：

任务先窄，闭环先深。

## 结尾

Agent 不是一次模型采购，也不是一组 Prompt 项目。

它是一套连接模型、工具、业务流程和组织责任的执行系统。

CTO 判断 Agent 项目是否健康，不要只看模型回答有多好。

要看系统有没有开始处理真实世界的问题：状态、权限、副作用、恢复和指标。

这些东西不够性感。

但它们决定 Agent 最后能不能上线。
