# 短会话稳，不代表能上线

封面文案：

- 主标题：短会话稳不算稳
- 副标题：Agent 长任务才是真考验

正文：

很多 Agent 项目在测试阶段都挺顺。

用户问一句，模型答一句；
调用一个工具，返回一个结果；
Demo 看起来很丝滑。

但一旦进入长会话，问题就开始出现：

上下文续不上、子任务结果丢失、工具调用状态不一致、用户刷新后进度没了、任务失败只能重头来。

这时候团队很容易误判：

“是不是模型不够强？”
“是不是上下文窗口不够长？”
“是不是要换更贵的模型？”

但如果短会话稳、长会话崩，问题大概率不在模型，而在 Harness。

长会话比短会话多了三类压力：

1. 状态压力：每一步产生的中间状态都要被追踪。
2. 恢复压力：任务中断后要知道从哪里继续。
3. 副作用压力：外部动作不能因为重试而重复发生。

模型可以生成一个看起来合理的计划，但它不会自动帮你解决“状态什么时候落盘”“工具结果怎么幂等”“子 Agent 超时后怎么降级”这些工程问题。

真正能上线的 Agent，不是一次回答很惊艳，而是长任务跑到一半失败后，系统还能清楚地知道：

已经完成了什么；
哪些动作不能重复；
哪些结果要重新拉取；
下一步从哪里继续。

这就是 Harness 的价值。

做 Agent 不要只测短链路 Demo。上线前至少跑一次“执行 -> 中断 -> 重启 -> 恢复继续”的闭环。

这条闭环跑不通，模型再强也只是 Demo 强。
