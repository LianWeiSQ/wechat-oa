# Harness 自检 4 问

封面文案：

- 主标题：Harness 自检 4 问
- 副标题：Agent 上线前先回答

正文：

如果你正在做 Agent 系统，先别急着问模型选型。

用这 4 个问题自检一下，很多项目会立刻暴露工程短板。

第一，状态是不是有唯一可信源？

一次 Agent 执行会产生很多状态：当前步骤、工具调用结果、子任务进度、错误信息、用户上下文。如果这些状态散落在多个对象里，没有一个 ThreadState / AppState 之类的单点真相，长任务一定会越来越难维护。

第二，失败是不是只有“成功/失败”两种？

真实系统里，子任务至少要区分等待中、运行中、已完成、失败、取消、超时。状态越粗，恢复策略越粗。最后就会变成：失败就重跑，重跑又带来重复副作用。

第三，工具调用有没有幂等设计？

发消息、下单、写数据库、调用外部 API，这些动作一旦成功，重试时就不能简单再来一遍。Harness 必须知道哪些动作已经提交，哪些可以安全重试。

第四，是否跑通过断点续跑？

不是文档上说支持，而是真的杀进程、重启、恢复，然后继续跑。这个测试很残酷，但它能快速看出系统是不是只有 Demo 能力。

这 4 个问题里，如果有两个回答不清楚，瓶颈大概率不是模型。

Agent 工程的难点，不是让模型说出聪明话，而是把不确定的模型行为装进一个可恢复、可追踪、可降级的执行系统里。
