# 工具调用成功了，状态却没记

封面文案：

- 主标题：状态没记，就会出事
- 副标题：Agent 工具调用的隐藏风险

正文：

Agent 上线后，有一类问题特别隐蔽：

工具调用其实成功了，但 Harness 没把结果记清楚。

比如：

邮件已经发出去了；
工单已经创建了；
数据库已经写入了；
支付或订单接口已经返回成功了。

但 Agent 的状态里只写了一个很模糊的字段：

“tool called”

这在 Demo 阶段没什么问题。可一旦任务中断、网络抖动、进程重启，系统就不知道这个外部动作到底是否已经完成。

接下来只有两个坏选择：

要么重复执行，可能造成重复发信、重复创建、重复扣款；
要么停止执行，让用户重新来一遍。

这不是模型问题，是 Harness 没有做好副作用管理。

工具调用必须至少记录：

1. 调用请求的幂等 ID。
2. 外部系统返回的结果 ID。
3. 这一步是否已经提交成功。
4. 失败时是否可以安全重试。
5. 恢复后应该跳过、重试还是人工介入。

Agent 工程和普通聊天机器人的差别就在这里。

聊天机器人说错了，可以道歉重说。

Agent 一旦连接真实系统，错误会变成真实副作用。

所以，别只问“能不能接工具”。更重要的是：工具执行后的结果，能不能被 Harness 稳定地接住。

话题标签：

#Agent开发 #工具调用 #AIInfra #系统设计 #后端工程 #幂等 #AI智能体

置顶评论：

做 Agent 工具层时，幂等 ID 和外部结果 ID 真的不能省。省掉的设计，最后都会变成线上事故。
