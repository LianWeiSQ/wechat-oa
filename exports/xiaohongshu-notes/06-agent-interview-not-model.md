# 面试 Agent，别只讲模型

封面文案：

- 主标题：别只讲模型
- 副标题：Agent 面试真正问的是工程闭环

正文：

现在面试 Agent 平台或 AI Infra，很多问题已经不是：

“你会不会调 GPT？”
“你用过几个模型？”
“RAG 怎么做？”

面试官会继续往下问：

一个 Agent 正在执行长任务，已经调用了两个工具，第三步网络超时，进程重启了。用户回来以后，系统怎么知道从哪里继续？

这时候如果你的回答还是：

“把聊天记录存下来。”
“失败了重新跑。”
“加个 try-catch 重试。”

基本就暴露了：你理解的是聊天机器人，不是生产级 Agent。

Agent 进入真实业务后，最怕的不是某句话答错，而是执行状态乱掉。

你至少要讲清楚 5 件事：

1. 有没有唯一可信状态源，比如 `ThreadState` 或 `RunState`。
2. 工具调用成功后，结果有没有写入 checkpoint。
3. 外部动作是否有幂等 ID，能不能避免重复执行。
4. 中断恢复时，系统怎么判断 completed、failed、timeout、unknown。
5. 状态未知时，是不是先对账，再决定重试或跳过。

真正拉开差距的点，是 unknown state。

比如 Agent 创建工单，请求发出去了，但响应没回来，网络断了。

这个工单到底创建成功了吗？

如果直接重试，可能创建两个。
如果直接跳过，可能漏掉一个。
如果让模型猜，模型也不知道真实世界发生了什么。

正确做法是用业务唯一键或查询接口先确认外部系统状态，再决定下一步。

所以，面试 Agent 平台时，不要只背模型选型。

更有说服力的回答是：

模型负责推理，Harness 负责把推理变成可恢复、可追踪、可审计的执行闭环。

这才是 Agent 从 Demo 走向生产的分水岭。

话题标签：

#Agent #AIInfra #大模型应用 #程序员面试 #系统设计 #后端工程 #AI智能体 #工程架构

置顶评论：

如果面试官问你“Agent 长任务中断后怎么恢复”，不要只说保存聊天记录。先讲状态源、checkpoint、幂等 ID、状态未知和恢复策略。
