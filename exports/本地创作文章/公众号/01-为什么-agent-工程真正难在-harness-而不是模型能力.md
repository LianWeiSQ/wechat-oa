## 一、一次中断恢复失败引发的排查

一个数据处理 Agent 被设计用来定时拉取多个数据源、清洗聚合、写入数仓。某个周五任务因上游 API 限流，在第二步批量写入时被操作系统 OOM Kill。Agent 框架宣称支持“断点续跑”，但周一重试时却发现：第一步重新拉取了全量数据，生成了重复的主键；第三步的部分输出未清理，导致下游任务收到脏数据。

问题不在模型——它正确理解了指令和重试策略。问题出在 Harness 层：状态保存的时机、恢复时对“已完成步骤”的判定、子任务进度同步与幂等保证，全部由编排系统决定。这次故障之后，团队才意识到：Agent 落地的工程瓶颈，很少是模型推理能力不够，而是“让 Agent 稳定、可恢复地完成长链任务”这件事，远比想象中复杂。

本文将讨论范围限定在这样一类 Agent：需要执行多步骤、可能调用外部工具、运行时间从分钟到小时级，且要求任务状态可恢复。不涉及前端交互、对话式 Agent 的流式输出优化，也不讨论强化学习微调。我们假设读者熟悉基本的 Agent 概念，如果还没有阅读过任何 Agent 编排框架源码，建议先跑通一个最小中断-恢复 Demo 再回来对照。

## 二、一句话判断：Agent 瓶颈不在大脑，在线束

> Agent 系统的工程核心不是“模型调度器”，而是“状态源唯一化 + 执行链路闭环”。

大多数讨论聚焦于模型如何规划（Plan）、如何反思（Reflection），仿佛有个更强的模型就能让 Agent 变得可靠。但在实际工程中，模型只负责产出“决策意图”，其本身既不能持久化一段任务进度，也不知道一个工具调用是否已经执行过。这些能力全部落在 Harness——一套负责接收入口、收敛状态、驱动执行循环、管理持久化与恢复的编排层。

Harness 一词借鉴自自动驾驶的“线束”概念：不是感知或算法本身，而是把传感器、控制器、执行器可靠连接起来的那套电线与接插件。在 Agent 系统中，它对应的是从用户请求到最终结果之间，那条串联模型、工具、记忆、存储的“执行总线”。

## 三、Agent 能力框架分层总览

为了清晰拆解 Harness 的职责，我们可以把 Agent 系统分为三层：

┌───────────────┐
│   入口适配层    │  ← CLI、HTTP、消息队列、定时器
├───────────────┤
│   会话编排层    │  ← Agent Loop、状态机、上下文管理
├───────────────┤
│  执行与持久化层  │  ← 工具调用、存储后端、恢复入口
└───────────────┘

- 入口适配层 ：负责将不同交互方式（命令行、API、事件触发）的请求，统一为一次标准化的会话启动。它不关心具体执行逻辑，只做参数解析、模式路由和上下文初始化。

- 会话编排层 ：Harness 的核心。它维护一个可信的会话状态对象，驱动 “思考-行动-观察-重复”（TAOR）循环，每一步都围绕该状态进行决策、执行和持久化。这里定义了 Agent 的生命周期：何时开始、何时暂停、何时重试、何时结束。

- 执行与持久化层 ：承接工具调用、管理记忆存储、提供可插拔的持久化后端（内存、文件、SQLite、PostgreSQL 等）。恢复时，编排层通过该层反序列化状态，使 Agent 从中断点继续。

这三层划分并非唯一标准，但它帮助我们把“Harness”的责任边界清晰标记出来：大部分关于“Agent 不可靠”的抱怨，都能映射到编排层或执行层的某个组件缺陷上，而入口层的变种只影响使用方式。

## 四、核心机制工程拆解

### 4.1 唯一可信状态源

定位 ：整个 Agent 系统的“心跳记录”，是保证恢复一致性的唯一依据。 职责 ：保存当前任务的全局进度、已完成的步骤列表、正在进行中的工具调用 ID、以及被暂停的上下文快照。 生命周期 ：会话创建时初始化；每完成一个步骤或进入等待时更新；恢复时反序列化并校验版本。 关键源码锚点 ：在参考实现 DeerFlow 中，对应 core/state/WorkflowState 类和 persistence/StateSaver 接口；Claude Code 的 Harness 中则体现为 AppState 全局对象。 依赖关系 ：编排层的状态机变更事件会触发 StateSaver 写入；恢复时由执行层的 Runner 读取最新快照。 容错作用 ：若状态丢失，Agent 只能从头开始；若状态写入不完整（如未记录子任务结果），恢复后可能产生重复操作。

### 4.2 执行循环与可恢复步骤

Agent 的经典循环 TAOR 只有加入状态检查点才能真正可靠。

- 思考（Think） ：模型根据上下文和状态快照生成下一步行动，此步骤无副作用，可以不持久化。

- 行动（Act） ：调用工具或执行代码，可能产生外部副作用。必须在行动前将意图状态记为“pending”，行动后根据结果更新为“completed”或“failed”。

- 观察（Observe） ：收集结果，并入上下文。此时持久化该步骤的完整记录。

- 重复 ：循环直到达到终止条件或人为暂停。

关键的工程细节是“幂等恢复点”：如果系统在 “Act” 阶段崩溃，恢复后根据状态中的 “pending” 记录，允许重放该行动（需保证行动本身幂等），或标记为需要人工介入。Harness 必须提供“至少一次”执行保证，而不是依赖模型去判断“我是否已经做过这件事”。

### 4.3 工具系统：定义、装配与执行

工具调用是 Agent 与外部世界交互的界面，Harness 通过工具注册表（Tool Registry）来管理能力供给。Claude Code 的设计将其拆为三层：能力定义（声明工具的名称、参数、描述）、装配（将工具按照权限、场景组合成可用的集合）、执行（实际发起调用、处理超时、重试和结果格式）。

对于恢复场景，Harness 必须能记录工具调用的输入和输出，以便恢复时重放或跳过。更严格的设计会要求每个工具声明其幂等性，例如“HTTP GET 天然幂等，POST 需要业务保证”。

### 4.4 记忆管理：不止是长上下文

很多人把记忆等同于上下文窗口，但 Harness 的记忆系统往往包含三层：

- 策略层 ：决定哪些信息需要被记住（如任务目标、用户偏好、关键中间结果）。

- 指令层 ：以系统提示或元数据形式嵌入模型输入。

- 存储层 ：文件、向量数据库或结构化记录，在恢复时重新加载。

Claude Code 中的记忆采用“LLM 驱动文件级选择”，即模型在决策时通过工具调用来读写特定记忆文件，Harness 提供稳定的文件访问接口并缓存索引。这种设计把“记什么”的决策权留给模型，但把“怎么持久化和恢复”的稳定性留给 Harness。

## 五、设计权衡与边界条件

没有完美的系统，只有清晰的取舍。以下是 Agent 工程中必须面对的典型权衡：

权衡点选择A选择B适用场景与风险
重复执行 vs 幂等保证要求所有工具幂等，安全重试允许非幂等，但跟踪执行记录A 实施成本高，B 在多次中断后可能产生重复记录
同步执行 vs 异步执行顺序执行，易于理解与调试并行子任务，提升吞吐同步状态一致性高；异步需解决并发状态合并，容易丢失更新
内存状态 vs 持久化状态纯内存，快速，但不可恢复全量持久化，可恢复，但性能开销开发调试可用内存模式；生产必须持久化，且要关注序列化性能
全局回滚 vs 局部失败一步失败则整体回滚允许跳过失败步骤，继续执行全局回滚浪费已完成工作；局部失败需定义清晰的“跳过”语义

这些权衡不是非黑即白，例如许多框架支持配置化切换后端（memory / sqlite / postgres）来在开发与生产环境间变化，但必须理解模式切换带来的语义差异，尤其是并发写入和恢复行为的细微不同。

## 六、项目落地与职业启发

### 对于想要验证想法的个人

- 用最小 Demo 跑通“开始执行—模拟中断—恢复执行”闭环。推荐使用 LangGraph 或类似的显式状态机框架，设置一个简单的两步序列，在第二步前强制退出，然后重新加载检查点。

- 阅读 LangGraph 的 Checkpointer 实现或 Claude Code 源码中 AppState 的序列化/反序列化部分，理解状态对象如何从持久化介质中复原。

- 列出你自己的场景中不可恢复的边界：涉及外部系统的非幂等写操作、无法回滚的副作用等。这会让你对 Harness 的能力极限有现实认知。

### 对于负责技术选型的团队负责人

评估一个 Agent 框架或平台时，不要只看它支持多少模型、有多少预置工具。花时间看它：

- 是否显式定义了任务状态的生命周期？状态对象是否可序列化？

- 是否提供可配置的持久化后端？切换后端是否需要改动业务代码？

- 在文档或源码中是否明确了恢复语义（至少一次、精确一次、手动干预）？

- 工具调用的超时、重试、幂等性是否有统一的抽象？

这些直接决定了它能否在生产环境的长期任务中存活。

### 对于准备 Agent 方向面试的工程师

当面试官问到“如何设计一个可靠的 Agent 系统”时，不要只回答怎么用 Prompt 让模型规划。你可以这样结构化：

- 先画分层图，说明 Harness 的核心作用。

- 重点讲状态管理和恢复机制，给出具体场景（比如断电后如何恢复一个正在进行中的工具调用）。

- 对比现有框架（如 LangGraph、AutoGPT）在这个问题上的不同取舍，展示工程判断。

这会比单纯罗列技术名词更有说服力，也更能体现你对系统边界的理解。

## 七、总结与未来展望

- Agent 工程的真正难点在 Harness 层，它负责状态的唯一化、持久化和恢复闭环。模型能力是必要但非充分条件。

- 一个可靠的 Harness 必须至少解决：状态模型定义、幂等执行、持久化解耦和恢复入口。缺少任何一个，都会导致线上故障。

- 设计上需要在重复执行、并发、持久化粒度之间做权衡，没有银弹，只有对场景的适配。

- 当前许多框架的恢复机制仍不完善，常见未完成功能如“部分恢复”“分布式状态合并”仍在源码的 TODO 中（推测未来会逐步完善）。

- 本文未涉及多 Agent 协作下的状态一致性问题、安全沙箱以及模型调度优化，这些领域同样值得探索，但需要深入分布式系统和安全相关基础。

如果你正在构建或评估 Agent 系统，不妨回到源码，找到那个最关键的状态对象，跟踪一次完整的“暂停-恢复”路径。这远比追逐模型更新更能提升工程内力。
