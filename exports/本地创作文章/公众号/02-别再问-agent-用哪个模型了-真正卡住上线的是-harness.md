# 别再问 Agent 用哪个模型了，真正卡住上线的是 Harness

最近帮一个团队复盘线上事故。他们的 Agent 在测试环境表现惊艳，切到最新强的模型后，对话质量明显提升。但上线不到一周，每次处理长时任务遇到网络抖动或工具调用超时，整个会话就直接挂掉——模型没变，Agent 却不可用了。

排查下来，问题出在那个一直被忽略的 Harness 层：任务中断后没有可靠的状态恢复，工具调用结果没有幂等保证，多步骤执行一旦失败就只能重头再来。更让人意外的是，团队里不少工程师甚至说不清楚 Harness 到底管什么。

这不是孤例。很多人在做 Agent 时，第一反应是“换个更强的模型”，却很少追问： 模型产出的结果，到底是怎么被组织成一次稳定的任务执行？中间断了怎么续？工具调用的副作用怎么消除？

## 一个反常识判断

> 模型决定了 Agent 的能力上限，但 Harness 决定了你能否稳定地触达那个上限。

Harness 不是调度器，也不是简单的胶水代码。它是一套负责 入口适配、会话成型、状态管理、执行循环和恢复闭环 的运行时。换句话说，Harness 是 Agent 系统的“操作系统”——没有它，模型推理出再完美的行动计划，也无法在现实环境中可靠地执行，更扛不住故障。

## Harness 到底管什么？三层收敛

把 Harness 拆开，它本质上在做三件事：

- 入口收敛 ：不论用户从命令行、Web 页面还是 API 调用过来，Harness 都将这些不同入口统一为同一套执行模型，避免为每个交互方式写一套状态逻辑。

- 会话成型 ：把单次的“用户输入 + 工具调用 + 模型输出”打包成一个可追溯、可中断、可恢复的 turn，并定义清楚每个 turn 的前置状态、执行过程和结果提交。

- 运行桥接 ：将模型、工具能力、持久化存储、消息队列等基础设施串联在一起，在真正执行 Agent Loop 时提供统一的上下文和恢复点。

你如果看过一些 Agent 框架的源码，会发现 Harness 往往是代码量最大的那层——它承载了所有“脏活累活”，而模型调用只是其中一个小小的环节。

## 状态管理：Harness 的“单点真相”

在 Harness 的设计里，有一个必须死守的原则： 唯一可信状态源 。一次 Agent 执行的所有关键信息——当前步骤、工具调用结果、等待重试的请求、已产生的副作用——都应该汇聚到同一个状态对象里，比如常见的 AppState 或 ThreadState。

这个状态对象不是只暂存内存，而是有明确的持久化路径。很多事故的根源，就是 Harness 层没有把这个“唯一真相”及时刷入存储，导致进程重启后上下文丢失。

我的判断是： 做 Agent 工程，你首先要确定的不是用 Claude 还是 GPT，而是你的 Harness 把状态存在哪里、多久存一次、谁负责更新、怎么回溯。

## 执行循环：不是模型一次调用，是 TAOR

Harness 里的执行循环通常是一个标准的 Think-Act-Observe-Repeat （TAOR）流程：模型根据当前状态思考下一步行动（Think），Harness 调用工具执行（Act），收集返回结果与外界反馈（Observe），然后更新状态并决定是否继续（Repeat）。

这里最容易踩坑的，不是模型思考错了，而是 工具执行的副作用没有纳入状态管理 。比如发了一封邮件，但状态里只记录“已调用发邮件工具”，却没有持久化发邮件的实际结果。一旦中断恢复，系统不知道邮件是否已发送，可能重复发送或漏发。

因此，Harness 在 Observe 阶段必须做结果结构化并存入状态，而不是只把原始返回丢给模型。

## 持久化与恢复：断点续跑不是魔法

所谓“断点续跑”，原理很简单：

- 每次 Agent 步骤结束后，Harness 将当前完整状态序列化并存入持久化后端（可以是内存、文件、SQLite 或 Postgres）。

- 进程恢复时，从存储中反序列化出最后一个成功状态。

- 重新进入执行循环时，Harness 会先检查未完成的工具调用、未提交的结果，然后从断点继续。

但工程现实比原理复杂得多：状态对象可能非常大，序列化开销影响性能；多人协作的分布式环境下，状态的一致性更难保证。Harness 必须在这中间做权衡，比如只持久化关键决策点，或者采用增量更新。

我观察到的成功实践是： 将状态拆分为“执行态”（内存中快速流转）和“持久态”（落盘的关键快照），Harness 负责二者之间的中转与对齐，而不是每次都全量落盘。

## 你在项目里应该关注的三个权衡

1. 重复执行 vs. 幂等保证 当任务中断后恢复，Harness 可能重复执行某个已成功的步骤。如果你调用的外部工具不支持幂等，就要在 Harness 层做去重或标记。

2. 在线实时 vs. 离线批量 长时 Agent 任务不一定需要全链路强实时。有时把执行挪到异步队列，由 Harness 管理生命周期会更可靠，但这要求 Harness 支持离线模式的恢复与进度查询。

3. 内存状态 vs. 持久化状态 一直在内存里跑当然最快，但崩溃全丢。持久化能恢复，但影响性能。Harness 需要根据任务类型动态选择策略，而不是全部一套方案。

## 给后端工程师的落地建议

如果你正在面试 Agent 平台研发或 AI Infra 岗位，面试官问你“设计一个可靠的 Agent 系统”， 重点不要放在选了哪个模型，而要讲清楚怎么管理状态、怎么保证工具调用可恢复、怎么实现执行闭环 。

项目里不妨先做一个最小验证：跑通“执行—中断（杀进程）—重启—恢复继续”的闭环。用最简单的模型，确保 Harness 层的状态管理和恢复机制正确。这个闭环跑通，比你集成三个大模型都有说服力。

然后带着这个闭环，去读你所用框架的 Harness 源码：找到状态对象的定义、持久化入口、恢复入口，理解它们之间的调用链。

## 结语

Harness 不是银弹，它也有力所不及的地方：极端并发下的状态一致性、多 Agent 协作时的部分失败、复杂依赖链的回滚，这些仍是开放问题。但它的确是 Agent 工程化的基石。

> 别再问 Agent 用哪个模型了。先把 Harness 层的状态管理、工具编排和恢复闭环做扎实——所有高级模型，最后都要站在这层地基上跑步。
