# 你以为是模型拖累了Agent，其实Harness才是真正的瓶颈

上周和一个做Agent平台的朋友聊项目进展，他说团队花了两周时间做模型对比测试，从GPT-4o到Claude 3.5到国产闭源模型跑了个遍，最后选了效果最好的那个上线。结果三周后灰度放量，第一批用户长会话跑了二十多轮，系统就开始出现状态错乱、子任务结果丢失、上下文无法续接的问题。

团队第一反应是模型不够强，要不要再加个KoPI加了长上下文窗口。但换个角度想：如果模型真的是瓶颈，那为什么短会话稳得很，长会话才崩？

这个问题我问过好几个在做的Agent项目的工程师，答案很一致——问题根本不在模型，而在Harness。

## 反常识：模型越强，Harness越重要

很多人理解里的Agent架构是“模型+Prompt+工具”，觉得只要模型够强，系统就能稳定运行。这是一种幻觉。

真实情况是：模型越强，它生成的中间状态越多、对工具调用的编排越复杂、长程推理的链路越长。而这些链路一旦缺乏有效的状态管理和断点续跑机制，系统就像一辆没有悬挂系统的跑车——动力越强，失控风险越高。

我更愿意把它理解为：模型负责“跑”，Harness负责“不翻车”。这两个能力的成长速度并不匹配，当行业都在追逐更强的模型时，真正的工程瓶颈其实在Harness层。

这不是概念洁癖，而是线上故障迟早会撞到的边界。

## Harness到底是什么，为什么它才是核心

简单说，Harness是Agent的“运行骨架”。它不负责生成内容，它负责让生成的内容按照正确的路径流转，并在出问题时有恢复的能力。

具体来说，一个完整的Harness需要处理四件事：

第一件事：状态编排。 Agent执行过程中会产生大量中间状态：子任务运行状态、子任务结果、全局上下文快照。这些状态在内存里怎么组织、什么时候同步到持久层、如何保证状态一致性，都需要Harness来定义规则。

以DeerFLow的多Agent执行为例，它定义了SubagentStatus枚举：PENDING/RUNNING/COMPLETED/FAILED/CANCELLED/TIMED_OUT。这个枚举定义了每种状态的语义和转换规则，ThreadState是主线程唯一可信的状态源，SubagentResult是子代理唯一的状态载体。Harness通过这套状态机规范了多Agent协作的行为边界。

第二件事：断点续跑。 这是Harness最考验工程能力的部分。当Agent执行到第十轮突然崩溃，系统怎么恢复到崩溃前的状态？需要恢复的不仅是上下文，还有当前运行到哪一步、子任务是否完成、哪些结果已经写入持久化层。

Claude Code的Harness在这一层做了会话持久化设计：sessionStorage保存全局状态，ThreadState保存会话状态，bootstrap保存初始化状态。这三层状态分离的设计让长会话恢复成为可能。

第三件事：工具能力封装。 工具是Agent的行动边界，但工具多了之后，谁来管理它们的生命周期、谁来保证工具调用不出状态泄漏、谁来处理工具超时和降级？这些问题模型回答不了，只有Harness能回答。

Claude Code的工具系统设计了三层：能力定义→能力装配→能力执行。这个链路的核心是标准化——把不同来源的工具封装成统一的执行契约，Harness才能统一调度。

第四件事：错误处理与降级。 模型会产生幻觉，工具会超时，状态会不一致。Harness需要设计错误处理的策略：什么时候重试、什么时候降级、什么时候终止会话反馈给用户。这些决策逻辑不是Prompt能解决的，它需要Harness层面的显式设计。

## 三个Harness设计的核心矛盾

理解了Harness是什么，接下来要搞清楚Harness设计难在哪里。根据我对几个开源Agent框架的源码研究，有三个核心矛盾始终存在：

矛盾一：状态实时性与持久化开销的矛盾。

状态需要持久化才能断点续跑，但频繁持久化会带来IO开销影响响应速度。DeerFLow的解法是双模式持久化：async_provider.py负责异步持久化，provider.py负责同步持久化。异步模式保证响应速度，同步模式保证关键节点的可靠性。

但这个双模式设计本身带来了复杂度：什么时候走异步、什么时候走同步、异步失败后怎么降级同步，这些边界条件都需要Harness来处理。

矛盾二：内存状态与持久化状态的一致性矛盾。

Agent运行时大量状态在内存里，但这些状态什么时候写持久化层、写的时候是否需要锁、多个Agent并发修改同一状态怎么保证一致性——这些问题在单机场景下不明显，在分布式场景下就是噩梦。

DeerFLow的设计思路是让ThreadState作为主线程唯一可信状态源，所有内存状态修改都需要经过这个“单一信源”，持久化是同步操作。这个设计牺牲了部分性能，但换来了状态一致性。

矛盾三：灵活性与规范性的矛盾。

Harness需要足够灵活以适配不同的Agent场景，但又需要足够规范以保证系统稳定。Claude Code的Harness设计了“入口适配+会话成型+运行桥接”的三层架构，这个架构的核心是把多入口、多模式、多运行位置收敛成统一的turn执行模型。

换句话说：Harness把复杂性封装在内部，对外暴露简单的接口。Agent开发者不需要关心请求从哪里来、运行在本地还是远端，只需要按照turn契约组织输入输出就行。

## 怎么判断你的Harness设计是否过关

给一个简单的判断框架：

- 能否在Agent崩溃后从断点恢复而不丢失已完成的子任务结果？
- 状态变更是否通过统一的状态管理层而非散落在各个Agent代码里？
- 工具调用失败后系统是否有预设的降级策略而非直接报错？
- 多Agent协作时的状态竞争是否有显式的锁或事务机制？

如果这四个问题里有两个以上的答案是“不太确定”，那你的瓶颈大概率在Harness而不是模型。

## 给真正在工程里踩过坑的人

写这篇文章的目的不是否定模型能力，而是想把注意力拉回到真正影响系统稳定性的地方。

模型能力决定Agent的上限，Harness决定Agent的下限。下限不够高，上限再高也没有意义。

如果你正在经历长会话崩溃、状态错乱、断点续跑失败这些问题，与其花时间做下一轮模型对比测试，不如先把Harness层的设计复盘一遍。答案往往不在更贵的模型里，而在更扎实的工程设计里。
